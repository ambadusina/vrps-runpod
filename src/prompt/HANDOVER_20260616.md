## soft_shift hole-fill left/right-eye direction fix

**Symptom**: realtime 2D->3D (and offline GPU soft_shift) showed asymmetric
distortion on foreground objects -- a face's edge stretched on the left eye and
clipped on the right eye -> ghosting / uncomfortable stereo in VR.

**Root cause**: `offline/two_dvr_gpu.py` `fw_fill` (the GPU forward-warp
hole-fill kernel) used one eye-independent "prefer the farther / smaller-near
neighbour" rule, with a tie-break that always picked the right neighbour. But the
two eyes warp in opposite directions (left eye pushes the foreground right, right
eye left), so the disocclusion gap opens on opposite sides and must be filled
from opposite sides. The CPU reference (`offline/two_dvr_render.py`, validated /
correct) fills per-eye with opposite fixed directions: left eye
`_shift_fill_holes_rgb(..., -1)` (left side), right eye `(..., +1)` (right side).
The GPU port lost that per-eye asymmetry, pulling the foreground into the gap on
one eye and clipping it on the other.

**Fix**: in `fw_fill`, when both neighbours are valid, pick by eye --
`pick = (eye == 0) ? li : ri;` -- mirroring the CPU direction. Single-sided
fallback unchanged. inverse_warp has no holes (pure inverse sampling) so it is
unaffected; the fix covers both flat3d and VR soft_shift (VR runs `fw_fill` then
`project_flat_lr`). The CPU path was already correct. Now that hole-fill is
user-selectable (`config.TWO_DVR_HOLE_FILL`, default soft_shift), this matters for
the default realtime preset.

Detailed report: `summary/summary_20260616_2DVR_SOFTSHIFT_HOLEFILL_FIX_CN.md`.

Verification:

- `python -m py_compile offline/two_dvr_gpu.py` clean. Change is a one-line
  ternary swap (+ comment), syntactically safe.
- **NOT** JIT-compiled / run on GPU here: sandbox bare python lacks `cv2`
  (environmental, per the 0615 handover) and no GPU. **Confirm in the real app**:
  run a 2D clip with a clear close foreground (face / near object) with hole-fill
  `soft_shift` and verify both eyes' edges are symmetric -- no one-side stretch /
  other-side clip.

Follow-ups / known limits:

- `fw_fill` still takes `nearmap/h/w` params (now unused; kept so `_launch`'s
  shared call signature doesn't change). Can be cleaned up later together with
  `_launch`. `_near_at` is still used by `fw_zbuf`/`fw_color`/`fw_blend`.
- If a smarter background-aware fill is desired later, do it *within each eye's
  allowed direction*, not as one cross-eye shared rule.

## soft_shift foreground-fattening (depth pre-smooth) fix

The per-eye direction fix above was necessary but NOT sufficient: realtime
soft_shift still fattened people -- one eye's object edge stretched/feathered
(foreground smeared into the hole), the eyes asymmetric ("face bigger / arms
thicker on one eye"). Diagnosed empirically in the uv env (cupy + cv2 + GPU)
with a synthetic foreground bar, dumping the forward-warp + fill per column.

Root cause: `_smooth_depth` (3x3 Gaussian on depth) runs *before* the forward
warp. The blurred depth discontinuity gives the boundary pixel an intermediate
disparity that scatters an isolated 1px foreground sliver into the disocclusion
gap, between the true background and the object. The directional hole-fill then
grabs that sliver (its nearest valid neighbour) instead of the revealed
background,摊开/fattening the foreground (bar 48px -> 52px). Sharp depth edges
keep the gap clean: unsmoothed -> fills from true background -> bar stays 48px.

Fix: new `offline/two_dvr_render.py:near_from_depth(depth, hole_fill_mode)` --
soft_shift normalizes the *un-smoothed* depth; inverse_warp keeps the blur (it
builds a continuous resample map where the blur suppresses edge stair-stepping).
Routed through it: `_make_soft_shift_pair`, `StereoRenderer` general path,
`GpuStereoRenderer.render()` (by `self._soft`), realtime
`pipeline/pynv_stream.py:_worker_loop_two_dvr`, offline
`offline/two_dvr_pynv.py:convert_clip_pynv`. The per-eye fill direction from the
previous fix is unchanged -- it just needs a sliver-free gap to work.

Verification (uv env, synthetic bar; correct stereo keeps width 48):
`soft_shift CPU [48,48] GPU [48,48]` (was 52), `inverse_warp [42,42]`
unaffected. `py_compile` clean; `tests/test_offline_outputs.py` 6 passed.
**Still confirm in the real app** (test_4k2d.mp4, hole_fill=soft_shift): both
eyes' faces/limbs equal thickness. Note the sharp "knife-cut" edge on the
*occlusion* side is correct (foreground over background), not a bug.

Detailed report:
`summary/summary_20260616_2DVR_SOFTSHIFT_DEPTH_SMOOTH_FATTEN_FIX_CN.md`.

## soft_shift sliver-smear root fix (depth-edge toggle sharpen)

The previous two fixes (per-eye direction, drop pre-smooth) were correct but
still insufficient on real REALTIME content: a person's edge had foreground
(skin) smeared into the disocclusion gap. Diagnosed in the uv env by rendering
the real `videos/test_4k2d.mp4` `00:48:00` person frame through actual DA3 depth
+ the realtime GPU `render_into_gpu`, inspecting pixels.

Root cause: DA3 depth at an object contour is a SOFT ramp (near goes 0->1 over
several px). The forward warp maps those intermediate disparities to foreground
slivers scattered across the whole ~33px disocclusion gap (65mm eye distance =
+/-33px disparity), and the 1D hole-fill grabs them regardless of direction.
Fundamental to soft_shift forward-warp (CPU, GPU, and original tool_2dvr alike);
the earlier fixes can't remove it because DA3 depth itself is soft. (inverse_warp
has no holes and was clean; user chose to keep soft_shift, so its fill was fixed.)

Fix: toggle (morphological-contrast) sharpen of the near map's horizontal edges,
snapping each pixel to the nearer of its local horizontal min/max -> the soft
contour becomes a hard edge IN PLACE (no net widening), so the gap collapses to a
clean hole bounded by solid fg/bg and fills from true background. Must be done at
FULL resolution (sharpening low-res near gets re-blurred by the kernel's bilinear
upsample).
- GPU (realtime + offline pynv + GpuStereoRenderer): inside `_near_at` in
  `offline/two_dvr_gpu.py` -- after bilinear, take local min/max over ~+/-6
  full-res px (a few low-res reads) and snap. No extra buffers/CPU/upload. Only
  soft_shift's fw_zbuf/fw_color use `_near_at`; inverse_warp/VR kernels untouched.
- CPU (offline ffmpeg fallback): `offline/two_dvr_render.py:_sharpen_near_edges`
  (cv2 erode/dilate 1xk + where), applied in `_make_soft_shift_pair` at full res.
GPU and CPU are separate paths -> no double sharpen.

Verified (uv env, real 00:48:00 person frame): realtime GPU render_into_gpu and
CPU StereoRenderer both produce a sharp body contour against the wall, skin smear
gone, eyes symmetric. py_compile clean; tests/test_offline_outputs.py 6 passed.
Confirm in the real app realtime mode (restart the server to load new code).

Detailed report:
`summary/summary_20260616_2DVR_SOFTSHIFT_SLIVER_SMEAR_FIX_CN.md`.

## soft_shift feather -> hard silhouette + background-only seam softening

Algorithm aligned with the user: (1) holes fill from background only, never the
moving foreground; (2) the fg/bg silhouette is a HARD edge, never feathered;
(3) only the seam between the stretched-background fill and the real background
is softened (real video backgrounds are complex, so keep the feather). Extreme
case (solid-color bg): hard fg/bg split + shift + paint bg color => no feather
needed anywhere; feather only hides texture-stretch of the copy-fill.

The old feather (fw_blend / _soft_blend_holes_rgb) blurred the whole 1px-dilated
hole boundary symmetrically, so it softened the silhouette too (edge looked
mushy). New approach: blend ONLY filled hole pixels, with the blur kernel
EXCLUDING foreground neighbours -> silhouette stays crisp, only the background
side is smoothed. Foreground is identified from the warped near (GPU: encoded in
zbuf as near*1e6+1; CPU: the warped near_buffer); the kernel gates the near half
only when the window straddles a depth step (near range > 0.30).
- GPU `offline/two_dvr_gpu.py:fw_blend`: only zbuf==0 pixels softened; K=3/V=2
  window; passthrough elsewhere. soft_shift only.
- CPU `offline/two_dvr_render.py`: `_forward_warp_eye_rgb` now also returns the
  warped near_buffer; `_soft_blend_holes_rgb(image, holes, near_buffer)` is a
  masked box blur `box(image*bg)/box(bg)` with `bg = hole | near<=thr`, blended
  0.35 on holes only; `_make_soft_shift_pair` wires it. `_dilate_mask_np` /
  `_box_blur_rgb` now unused (left in place).

Verified (uv env, real 00:48:00 person frame): realtime GPU body contour is crisp
against the wall, background fill softened, no mushy silhouette; CPU smoke OK.
py_compile clean; tests 6 passed. Restart the server to load in realtime.

Full soft_shift pipeline now: near_from_depth (no pre-smooth) -> toggle sharpen
(hard depth edge) -> forward warp + z-buffer + per-eye background-side fill ->
foreground-excluded feather (hard silhouette, soft bg seam). inverse_warp remains
the hole-free alternative.

Detailed report:
`summary/summary_20260616_2DVR_SOFTSHIFT_HARD_SILHOUETTE_FEATHER_CN.md`.

## soft_shift residual skin smear root fix (foreground near dilation)

Reproduced by generating test_4k2d 30-60s OFFLINE with current code and pulling
frames from the ENCODED mp4 (34-37s): a vertical skin band still smeared into the
person's left-edge gap. Deeper cause than the soft depth edge: the depth boundary
never aligns exactly with the colour silhouette, so a ring of object-COLOURED
pixels gets a low/intermediate near, does NOT travel with the foreground, and
stays put as a foreground-coloured sliver on the BACKGROUND side of the gap. The
directional fill grabs it. These slivers have LOW near, so near-based foreground
exclusion (toggle, the feather gate) can't catch them.

Fix: grow the foreground in the near map (`_dilate_near_fg`, a max filter) so the
boundary ring travels WITH the foreground; the pixels left at intermediate near
are now background-coloured, so any residual sliver blends invisibly into the
filled background. Radius auto-scales with near-map width (r=round(w/512): 518->1,
1920->~4, 4K->~8). Applied for soft_shift only:
- `offline/two_dvr_render.py:_dilate_near_fg`; `near_from_depth` soft branch
  dilates (covers realtime pynv_stream + offline two_dvr_pynv, both via
  near_from_depth); `_sharpen_near_edges` dilates before the toggle (CPU).
- `offline/two_dvr_gpu.py:GpuStereoRenderer.render()` soft branch dilates; the
  in-kernel `_near_at` toggle still hardens the dilated edge (no double-dilate:
  GPU near comes pre-dilated from near_from_depth).

Verified (uv env): regenerated 30-60s offline (TensorRT depth, pynv-gpu, ~83fps);
left-eye crops at 34/35/36/37s from the encoded mp4 show clean, crisp body
contours against the wall, skin smear gone, consistent across frames. tests 6
passed; py_compile clean. Comparison clip left at
videos/test_4k2d_S000030_E000100_3D_LR_Screen.mp4.

Full soft_shift chain now: near_from_depth (no pre-smooth) -> _dilate_near_fg
(grow fg, push boundary-mix onto bg colour) -> toggle/_near_at (harden edge) ->
forward warp + z-buffer + per-eye background-side fill -> foreground-excluded
feather (hard silhouette, soft bg seam). inverse_warp remains the hole-free
alternative.

Detailed report: `summary/summary_20260616_2DVR_SOFTSHIFT_FG_DILATE_CN.md`.

## soft_shift hair bands RESOLVED -- hybrid hole fill landed on GPU

## hybrid: background-side rim cleanup (matting leftover)

Follow-on requested by the user: matting/depth leaves object-coloured pixels
(hair/cloth) at the foreground's original edge with LOW depth, so they don't
shift and form a contaminated rim on the BACKGROUND edge of the gap (not a hole,
so the plain hybrid leaves them). Extended `fw_hole_from_inv` to also pull those
from inverse_warp, with the user's two safeguards:
- directional: only the eye's background side (left eye = a pixel with a gap to
  its right; right eye = gap to its left) -> never touches the foreground
  silhouette edge of the gap;
- depth-gated: only near < 0.5 (background/sliver); foreground (near >= 0.5) is
  kept, which also protects thin foreground strips between narrow gaps from being
  eaten.
Width `rim` scales with src width (1080p->8px, 4K->16px), env `PT_TWO_DVR_RIM`
overrides (0 disables). Measured rim0->16 changes ~0.7-3% px/frame, a few hundred
of them large (real sliver removal); the rest is background-edge pixels swapped
for near-identical inverse (harmless, reduces the rim's frame-to-frame flicker).
flat3d soft_shift only.

Verified (uv env): 30-60s offline regen ~83 fps; tests 11 pass. Clip refreshed:
videos/test_4k2d_S000030_E000100_3D_LR_Screen_HYBRID.mp4. This rim step
intentionally changes some NON-hole pixels, a deliberate superset of the
pure-hybrid prototype (test_two_dvr_hybrid_hole_fill tests the prototype function,
not the kernel, so it still passes).

  Without this, HD models would still get 518 input.

## 2DVR user-facing 3D strength

Added a user-facing `3D strength` control for 2D->3D and removed eye-distance
from the UI. The strength is a multiplier over the existing 65mm/default
disparity baseline: 100% keeps the old behavior, lower/higher values reduce or
increase the maximum horizontal disparity. Render code still clamps final pixel
shift through the existing `_max_disparity_pixels` limit, so the UI control does
not bypass the per-resolution max-disparity guard.

Backend:
- `offline/two_dvr_render.py`: added `DEFAULT_STRENGTH`,
  `strength_multiplier()`, and `effective_eye_distance_mm()`.
- Offline CLI accepts `--strength`; `--eye-distance` remains for script/backward
  compatibility, and the effective baseline is `eye_distance * strength`.
- Offline ffmpeg renderer, offline PyNv renderer, and realtime PyNv renderer all
  use the same effective-eye-distance helper.
- Realtime config now accepts `PT_TWO_DVR_STRENGTH`; logs include both strength
  and the resulting effective eye distance.

UI:
- Offline 2DVR page replaces the `瞳距/mm` controls with a `3D强度` dropdown
  (`50/75/100/125/150/175/200%`) and sends `--strength`, not `--eye-distance`.
- Realtime home page no longer has the 2DVR Configure dialog/button. The 2DVR
  row exposes only the on/off switch plus the `3D强度` dropdown immediately after
  it.
- Realtime hidden settings are fixed to product defaults when saved/emitted:
  model `base`, hole fill `soft_shift`, eye distance `65.0`; only strength is
  user-controlled from the UI. Old saved eye-distance values migrate to an
  equivalent strength once, then eye distance is reset to 65.

## realtime 2DVR DA3 engine prewarm

Investigated `debug_output/server.log`: realtime 2D->3D steady-state was already
~30fps because the source/output FPS was 29.97 and producer pacing was active;
the compute stages were ~11-12ms/frame (roughly 80fps throughput). The visible
startup penalty was the DA3 session path: cache hit was logged, but startup did
not actually create a `Da3DepthEngine`, so the first realtime request still paid
~7.3s for engine/session creation plus a ~556ms first depth inference.

Fix:
- `offline/da3_depth.py`: added `warmup_depth_engine()`, a retained in-process
  DA3 engine cache keyed by `(model, provider)`. It creates `Da3DepthEngine`,
  runs one dummy `session.run`, stores the engine, and returns the same warmed
  object to later callers.
- `main.py:_warmup_da3_trt_if_needed`: even when the TRT engine cache already
  exists, startup now warms and retains the selected realtime DA3 preset instead
  of returning after `trt_engine_cached()`.
- Startup DA3 prewarm is non-fatal: `main.py:_warmup_da3_trt_nonfatal` logs a
  warning and continues if the retained-engine warmup fails. This keeps
  green/alpha/DLNA startup available; the first 2D->3D request will lazy-load
  DA3 if startup prewarm was skipped/failed.
- `pipeline/pynv_stream.py` realtime 2DVR worker now calls
  `warmup_depth_engine()` instead of directly constructing `Da3DepthEngine`, so
  a startup-warmed session is reused by the first client request.

Expected effect: first realtime 2D->3D request should no longer show the
multi-second `two_dvr_da3_created` gap or the large first-frame TRT/ORT warmup
inside the request. Steady-state FPS is unchanged; it remains capped by the
source/output FPS and realtime pacing.

  `main.py`, `pipeline/pynv_stream.py`, and `tests/test_da3_download_and_trt.py`.
