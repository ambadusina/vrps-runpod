## Skybox live compatibility default

- Investigated the SkyboxVR loading regression against earlier handovers.
- Earlier stable Skybox behavior expected one real live producer plus duplicate `/passthrough_live` requests joining the same `LiveSession`.
- The 2026-05-28 multi-concurrency change made `PT_PASSTHROUGH_MAX_CONCURRENT` default to `auto`, which resolves to `2` on the current RTX 5060 Ti 16GB machine.
- Current `server.log` showed Skybox/libmpv request storms hitting `active=2 max=2` and returning `503 busy`.
- Restored the default `PT_PASSTHROUGH_MAX_CONCURRENT` to `1`; users can still set `PT_PASSTHROUGH_MAX_CONCURRENT=auto` explicitly for multi-stream testing.
- User retested with default `1` and reported Skybox still failed, so the default was restored to `auto` for continued diagnosis.

## Skybox versus 4XVR live request comparison

- Rechecked `debug_output/server.log` and `debug_output/request_history/request_history_20260531.jsonl` after the `MAX_CONCURRENT=1` retest.
- The retest process still logged `MAX_CONCURRENT=1`, so the failure is not explained only by the default `auto` resolving above one.
- Latest Skybox/libmpv playback for `sivr00170_B.mp4` sent a burst of `/passthrough_live` zero-open `Range: bytes=0-` requests across many chapter times (`t=1440,900,1080,1260,720,0,180,...`).
- Request history showed `route_profile=libmpv` had 39 live requests, all zero-open range; 25 returned `200`, 14 returned `503`.
- 4XVR/Quest history on the same day showed `route_profile=4xvr` had 26 live requests; all returned `200`, with a mix of no-range and nonzero byte range requests.
- Code comparison explains the behavioral split: `4xvr` and `avpro` managed live requests allow same-owner preemption, while `libmpv`/Skybox does not. Skybox only reuses an existing session when the live key matches exactly, so cross-time chapter probes contend for the active slot and can wait 10s before `503`.
- Do not simply enable broad same-owner preemption for `libmpv`: earlier handovers record that broad Skybox preemption could break duplicate startup requests. A safer fix would need Skybox-specific handling that preserves same-key duplicate subscription while collapsing or superseding cross-time startup probes.
