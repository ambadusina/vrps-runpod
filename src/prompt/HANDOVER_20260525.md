## macOS Compatibility Investigation

- User asked to evaluate what is needed for current project macOS compatibility.
- Current project state:
  - The UI/DLNA/HTTP layer is mostly Python/PySide6/FastAPI and can be made cross-platform.
  - The realtime production path is NVIDIA/CUDA-first and Windows-packaging-first.
  - `pyproject.toml` hard-depends on `onnxruntime-gpu`, `cupy-cuda12x`, `pynvvideocodec`, and `tensorrt-cu12`.
  - `config.py` defaults to `CUDAExecutionProvider`, `PT_USE_PYNV=1`, `hevc_nvenc`, and `PT_HWACCEL=cuda`.
  - `build_exe.py` assumes `.venv\Lib\site-packages`, `.venv\Scripts`, `.exe`, `.dll`, `.pyd`, CUDA Toolkit, cuDNN, TensorRT DLLs, and PyNvVideoCodec `.pyd`.
  - UI process helpers assume Windows venv and frozen executable names such as `python.exe` and `pt_core.exe`.
  - Windows Firewall setup is correctly guarded by `sys.platform == "win32"` and can become a no-op on macOS.
- External dependency check:
  - `onnxruntime-gpu` PyPI wheels are Windows x86-64 and manylinux x86-64, not macOS.
  - PyNvVideoCodec official docs list Windows, Linux, and WSL support, not macOS.
  - TensorRT installation docs list Linux and Windows platform support, not macOS.
  - CuPy CUDA wheels are for Linux/Windows and require NVIDIA CUDA; Apple Silicon GPU cannot run the CUDA/CuPy path.
- Conclusion:
  - macOS compatibility is feasible only as a reduced or alternative backend, not by directly porting the current NVIDIA production path.
  - A practical macOS MVP should support desktop UI, media library, DLNA browsing, thumbnails, source-file serving, subtitle preview, and possibly offline/slow CPU or CoreML/Metal inference.
  - Realtime 8K passthrough with the current PyNv/NVENC/TensorRT/CuPy design is not portable to macOS.
- Recommended implementation sequence:
  1. Split dependencies into platform extras/groups: Windows NVIDIA full stack vs macOS core stack.
  2. Add a platform capability module to centralize `is_windows`, `is_macos`, `has_cuda_stack`, `has_pynv`, `has_trt`, and available ffmpeg encoders.
  3. Change macOS defaults to `PT_USE_PYNV=0`, `PT_ONNX_PROVIDERS=CPUExecutionProvider` or a future CoreML provider, `PT_HWACCEL=none` or VideoToolbox for FFmpeg-only paths, and `PT_VCODEC=hevc_videotoolbox` only where FFmpeg supports it.
  4. Gate or hide TensorRT/PyNv/NVENC UI controls when the backend is unavailable.
  5. Make process and packaging helpers platform-aware: `.venv/bin/python`, executable suffixes, app bundle/icon formats, and PyInstaller data separators.
  6. Add CI/test matrix simulation by patching `sys.platform` for macOS defaults, process helper paths, route fallback, and UI visibility.
- No code changes were made for this investigation.
