# 2026-06-01 Handover

## Skybox `/passthrough_live` 长期"loading 不动"根因与修复

### 现象

- Skybox/2.0.x 打开 `/passthrough_live/<file>.mp4?...` 后 UI 一直 loading，从不开始播放。
- 同一台 Quest3、同一根 Wi-Fi、同一份 8K HEVC alpha live 流，**4XVR/HereSphere 能正常播**。
- Skybox 内部 debug overlay 显示这个请求"**网络流量为 0**"，pipeline 显示 `basic`。
- 服务端日志 + pcap 双方都看到字节实际发出去了（30fps 实时，~28 Mbit/s）。
- 老版本 Skybox 也复现，"以前能播"的状态早于某次 Skybox 客户端更新。

### 真正的根因

Skybox HTTP 层**按 URL 文件后缀选 playback pipeline**，而不是按 `Content-Type`：

- URL `/passthrough_live/<...>/movie.mp4?...` → Skybox 看到 `.mp4` 选 `pipeline=basic`（MP4 byte-range 解析）。
- 实际响应 body 是 MPEG-TS 字节，MP4 parser 找不到 `ftyp`/`moov` atom。
- Skybox HTTP 层把整个响应当成"非 MP4"丢弃，**字节根本不进入 decoder**——这就是 debug 显示零流量的原因。
- 4XVR/HereSphere/nPlayer/VLC 按 `Content-Type: video/MP2T` 选 pipeline，忽略 URL 后缀，所以正常。

之前所有围绕 DLNA flags、preempt、send pacing、duplicate subscriber 的修复都"打偏"了——服务端字节一直是好的，是 Skybox 的接收端在 HTTP 层就拦截掉了。

### 修复（关键的那一条）

DLNA 广播的 passthrough_live URL **末尾追加 `.ts` 后缀**，让 Skybox 切到 TS pipeline；路由侧把 `.ts`/`.m2ts`/`.mpegts` 当成"客户端管线提示后缀"，在文件查找前剥掉。

- [`dlna/content_directory.py`](../dlna/content_directory.py) — `_live_items` 和 `_live_chapter_items` 生成的 URL 末尾加 `.ts`。
  - 旧 URL: `/passthrough_live/.../movie.mp4?t=0&mode=alpha&ptv=9`
  - 新 URL: `/passthrough_live/.../movie.mp4.ts?t=0&mode=alpha&ptv=9`
- [`http_app/routes_media.py`](../http_app/routes_media.py) — 新增 `_strip_live_route_hint_suffix()` 和常量 `_LIVE_ROUTE_HINT_SUFFIXES = (".ts", ".m2ts", ".mpegts")`，`passthrough_live_get` 入口处先剥掉后缀再 `_safe_video_path()`。
- 合规客户端（4XVR / HereSphere / nPlayer / VLC）按 `Content-Type` 走，URL 后缀对它们无害。
- 旧的 DLNA 缓存 URL（没有 `.ts` 后缀）仍然能解析，向后兼容不破。

**Skybox 测试需在客户端里刷新 DLNA 列表才能拉到新 URL。**

### 期间走过的弯路（保留作为反面参考）

| 修复尝试 | 结论 |
|---|---|
| `MAX_CONCURRENT=1` / 调整 `BUSY_WAIT_SEC` | 无效，根本不是 slot 问题 |
| libmpv 加入 `allow_same_owner_preempt` | 反而引发 cascade 杀流，需要再加"不杀已建好的真实流"保护 |
| 把 SKYBOX UA 重路由到 4xvr profile | 没用——profile 决定的 header / 节奏 Skybox 根本不看 |
| 删掉 `TimeSeekRange.dlna.org`、改 `DLNA.ORG_OP=00`、`FLAGS=01700000`（"装 live"） | 没用——Skybox HTTP 层不看这些 |
| 让 duplicate subscriber 也流持续 chunk | 反向回归：3 路抢 Wi-Fi 把 primary 拖到 17fps |
| libmpv 截图探测从已有 session cache 取字节 | 反向回归：9 个探测 270MB 灌进 Wi-Fi 挤垮主播放 |

**教训**：当客户端 debug 显示"零流量"而服务端 pcap 显示有字节发出，**必然是接收端在 HTTP 层 reject**，不要再去服务端的 stream 内容里找问题。

### 顺带保留的有效改动

这些是真正解决了其它已知问题的改动，已合并保留：

1. **裸 `libmpv` UA = Skybox 章节截图探测** → 路由入口秒回 `503`，不占 slot 不启 GPU 不消费带宽。匹配方式 `is_libmpv_screenshot_probe_ua`：严格 `ua.strip().lower() == "libmpv"`，避免误伤真 mpv 或带版本号的 SKYBOX UA。
2. **`_take_active_slot` 的 same-owner preempt 仅作用于 `object()` slot_token**，不杀已 build 出的 `PyNvPassthroughStream` 或 `LiveSession`。判定 `_is_real_active_stream = callable(getattr(stream, "close", None))`。
3. **libmpv profile 同 `live_key` 启动期 debounce**（扩展自原 nPlayer 路径），保护 race 到 `_take_active_slot` 的同 key 重复请求。

### 还原回去的临时改动

- SKYBOX UA → 4xvr profile 的重路由（task #11）。SKYBOX 现在依然走原来的 libmpv profile。
- 给 SKYBOX UA 单独 strip TimeSeekRange/估算大小/改 OP=00 的"伪装 live"改动（task #10）。所有 managed live session 共用同一套 DLNA header。
- duplicate subscriber 持续流（task #7）。重复 subscriber 仍然只发 snapshot + EOF，保持 2026-05-10 的 backpressure 隔离。

关键 regression 测试：
- `test_strip_live_route_hint_suffix_drops_ts_for_skybox_pipeline` — `.ts` 后缀剥离正确，向后兼容无后缀的旧 URL。
- `test_libmpv_screenshot_probe_returns_503_when_no_producer` — 截图探测秒 503，断言 `_take_active_slot` / `_select_passthrough_stream` 都没被调用。

## DeoVR CDS passthrough 入口不可见修复

- 用户提供 Wireshark 抓包：DeoVR 的 `/control/cds` Browse 响应中实际包含 `*_passthrough_live` item，但播放器列表只显示普通真实视频。
- 抓包中的 DeoVR Browse 指纹：
  - `User-Agent` 为空；
  - `RequestedCount=0`；
  - `Filter=res,res@size,res@duration,dc:date,upnp:albumArtURI`。
- 判断：这是客户端展示层过滤 CDS item，不是服务端没有返回 item。当前默认 passthrough live `res protocolInfo` 为 `video/MP2T + HEVC_TS_NA_ISO + DLNA.ORG_OP=00 + DLNA.ORG_CI=1`，且省略 `DLNA.ORG_FLAGS`、`duration`、`bitrate`；DeoVR 对这类 live/limited TS 元数据比其它播放器更严格。
- 修改：
  - `http_app/routes_dlna.py`
    - SOAP 字段提取新增 `Filter`。
    - 新增 `_cds_client_profile()`，仅在明确 DeoVR UA（`deovr` / `[DEO...]`）或上述空 UA Browse 指纹下返回 `deovr`。
    - 将 `client_profile` 传给 `dlna.content_directory.handle_soap()`，并把 `cds_client_profile=deovr` 写入 request-history annotation。
  - `dlna/content_directory.py`
    - 默认行为不变：其它播放器仍看到 Skybox 兼容的 `.ts` route hint，以及 `OP=00`、无 flags/duration/bitrate 的 live 条目。
    - 仅 `client_profile="deovr"` 时，passthrough live URL 恢复为无 `.ts` 后缀的 `/passthrough_live/<file>.mp4?...`。
    - 仅 `client_profile="deovr"` 时，live `protocolInfo` 恢复旧式 `DLNA.ORG_OP=10;DLNA.ORG_CI=1;DLNA.ORG_FLAGS=41700000000000000000000000000000`，并保留 `duration` 与 `bitrate`。
    - 目录 item cache key 加入 `client_profile`，避免 DeoVR 兼容 DIDL 与其它播放器默认 DIDL 互相污染。
- 测试：

  - 新增 `test_deovr_live_chapter_items_use_legacy_cds_shape`。
  - 新增 `test_deovr_cds_profile_detects_blank_ua_browse_fingerprint`。
