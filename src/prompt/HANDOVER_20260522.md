## Offline CUDA vs TensorRT timing interpretation

- User compared offline CUDA and TensorRT runs on `videos/test_8k.mp4`:
  - CUDA: `decode_avg=8.706ms`, `matting_avg=19.220ms`, throughput `35.08fps`;
  - TensorRT: `decode_avg=20.392ms`, `matting_avg=6.476ms`, throughput `36.50fps`.
- Interpretation:
  - the offline `dec` timer wraps `dec.frame_at(src_idx)`, so it measures wall-clock wait until the next decoded frame is available, not pure NVDEC hardware execution time;
  - when CUDA RVM is slow, the matting step gives NVDEC more time to advance before the next `frame_at()` call, so `dec` appears smaller;
  - when TensorRT RVM is fast, the main loop returns to `frame_at()` earlier and waits for decode, so `dec` appears much larger;
  - `decode + matting` is the better quick comparison: CUDA is about `27.9ms`, TensorRT is about `26.9ms`, which matches the small throughput gain.
- Code changes:
  - `tools/offline_passthrough.py` and `tools/offline_alpha_passthrough.py` progress lines now include `dec+mat=...ms`;
  - summaries now include `decode_matting_avg = ... ms`;
  - static TensorRT session creation now sets ORT `log_severity_level=3` unless TRT subgraph dump/detailed build logging is enabled, reducing normal TensorRT initializer warnings in offline logs.
  - `pipeline/matting.py` also sets ORT process default logger severity to error level in normal mode so TensorRT EP logger warnings such as `WeightsContext.cpp:469` do not flood offline output; detailed logs remain available when `PT_ONNX_TRT_DUMP_SUBGRAPHS=1` or `PT_ONNX_TRT_DETAILED_BUILD_LOG=1`.

## 2026-05-22 4K alpha live FPS cap investigation

- User reported 4K alpha live can run around `100fps` with unlimited FPS, but drops below `30fps` when output FPS is capped to `40`.
