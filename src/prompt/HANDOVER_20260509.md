- User reported that switching to `PT_PASSTHROUGH_AUDIO_MPEGTS=aac` made video fail to display.
- Server log finding with `[180 VR] STAYC - RUN2U.mp4`:
  - `audio=aac` started quickly and emitted the first chunk after about 2 seconds;
  - output then stalled at about 109 KiB while the producer continued generating frames;
  - the route stall watchdog closed the stream after 8 seconds without response-byte growth.
- Tool-level reproduction using the same file copied into `videos/`:
  - normal AAC mux variants emitted only a small TS prefix and then stopped growing;

- Updated `pipeline/pynv_stream.py`:

## 2026-05-09 live 内部生产限速拆分

  - `LiveSession._run()` 也遵循 `PASSTHROUGH_REALTIME_PACING`，关闭时不再按 `output_fps` sleep。

## 2026-05-09 live pacing 拆分：生产端性能与发送节奏分离

- 用户明确核心目标：PyNv/GPU 路径必须保留 30fps 以上的生产余量，用于后续换模型和实验；不能只把 live pacing 改回 30fps，使输出能力退回到 FFmpeg 时代。
- 问题确认：
  - 旧 `PT_PASSTHROUGH_REALTIME_PACING` 同时控制 `pipeline/pynv_stream.py` 的 PyNv worker sleep 和 `http_app/routes_media.py` 的 LiveSession 发布 sleep；
  - 设为 1 会把 `interval_fps` 锁到 30fps 左右，掩盖真实生产能力；
  - 设为 0 又会把 100fps+、180-200Mbps 的 MPEG-TS 洪峰直接灌给播放器，出现 FPS 过快但画面卡住。
- 修复方向：
  - 新增 `PT_PASSTHROUGH_SEND_REALTIME_PACING=1`，只控制 HTTP live 发送节奏；
  - 新增 `PT_PASSTHROUGH_PRODUCER_REALTIME_PACING=0`，只控制 PyNv worker 是否按 wall-clock sleep，默认关闭以保留生产端 headroom；
  - `PT_PASSTHROUGH_REALTIME_PACING` 保留为旧兼容别名，仅作为 send pacing 默认值。
- 代码变更：
  - `pipeline/pynv_stream.py` 改为只看 `PASSTHROUGH_PRODUCER_REALTIME_PACING`；
  - `LiveSession._run()` 去掉按输出 FPS sleep，使其尽快读取 PyNv/mux 输出；
  - `LiveSession.subscribe()` 对缓存快照拆成 64KiB 小块并按估算码率发送；
  - VLC/default 直接路径增加有界发送队列，producer 可以短时间超前，队列满后才反压；
  - subscriber 队列满时不再直接丢客户端，而是给 producer 施加背压，符合单用户播放模型。
- 验证：
  - `python -m compileall config.py http_app\routes_media.py pipeline\pynv_stream.py` 通过。
- 后续观察：
  - PyNv 日志中的 `fps/interval_fps` 应重新反映生产能力，理论上可高于 30fps；
  - live response 日志会输出 `send_bps` 和 `send_pacing=True`；
  - 如果播放器仍冻结，应优先调发送端估算码率/突发窗口，而不是重新给 producer 加 30fps sleep。
