## Follow-up Fix: Offline Stop Leaves Child Processes

- User reported stopping 2048 MatAnyone2 offline conversion marked the UI log as stopped, but GPU memory remained allocated and a Python process was still running.
- Root cause: `OfflineProcess.stop()` only terminated/killed the direct UI-launched parent process. Offline conversion can spawn child processes:
  - `offline/convert.py` -> `tools/offline_passthrough.py`
  - MatAnyone2 prepass child process
  - FFmpeg audio/video helper processes
- Fixed `ui/services/offline_process.py` to mirror server shutdown behavior:
  - terminate direct process
  - wait 3s
  - on Windows run `taskkill /PID <pid> /T /F` to kill the whole process tree
  - wait 5s
  - fall back to direct parent kill
- This applies to all offline engines, including RVM and MatAnyone2, because they share `OfflineProcess.stop()`.
- Added tests in `tests/test_process_services.py` for successful and failed offline `taskkill` paths.

- Interpretation:
  - Scheme A is a real improvement over old no-band G2 and is safe enough to keep as an experimental switch.
  - It is not yet clean enough for default-on because segment-start frames can still get visible edge expansion. This is likely still guided-filter alpha growth inside the allowed confidence band, not the previous unbounded halo.
  - Next candidate should be a stricter G2 gate or a different refinement rule, e.g. morphology trimap band, signed max-delta around base (`base +/- delta`, not only upper clamp), or mask-aware weighted guided filtering.
