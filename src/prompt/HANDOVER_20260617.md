## 2DVR White Vertical Stripe Follow-Up

User reported that hybrid hole fill is mostly OK, but white vertical stripe
noise remains inside/near the person in offline output
`videos/test_4k2d_S000030_E000130_3D_LR_Screen.mp4`, especially output 3-9s
(source 33-39s). User's observation that left/right-eye stripe positions are
not symmetric was important: it means the artifact is not simple source matting
residue.

Findings:
- There are two visually similar but different artifacts:
  - foreground/body-internal white vertical dashes; and
  - disocclusion hole/background seam lines on the background side of hair.
- The foreground/body-internal issue is not primarily a RIM problem. Many visible
  white/cyan vertical dashes are forward-warp
  artifacts inside the body:
  - `zbuf==0` foreground-internal cracks later filled from inverse_warp, and
  - `zbuf!=0` but low-near/background-coloured written pixels embedded in the
    foreground region.
- Pure `zbuf==0` hybrid hole fill cannot distinguish those foreground-internal
  cracks from real disocclusion holes, and low-near written pixels are not holes
  at all.
- The later BASE518 report (left eye, hair left-side hole/background boundary,
  output 3-10s / source 33-40s) is the second class: a background-side seam next
  to a large true disocclusion hole. `PT_TWO_DVR_FG_BAD=0` vs `8` barely changes
  that seam, while `PT_TWO_DVR_RIM=8/16` visibly cleans it; `16` was cleaner than
  `8` in the t=3-10 focus sheet.

Implemented in commit `eaf5066`:
- Added `fw_fg_bad_local` inside the existing `_SOFT_SHIFT_KERNELS` RawModule in
  `offline/two_dvr_gpu.py`.
- It is controlled by `PT_TWO_DVR_FG_BAD`; default is auto-scaled
  `max(2, round(src_w / 240))` (`1920w -> 8`, `3840w -> 16`), and
  `PT_TWO_DVR_FG_BAD=0` disables it for diagnostics.
- When enabled, it replaces a background-ish pixel (hole or low-near written
  pixel) only if high-near foreground exists on both horizontal sides within the
  configured window in the same eye. This avoids touching true silhouette-side
  background, which has foreground on at most one side.
- The kernel runs after `fw_blend` and before VR projection, so downstream code
  does not re-soften the corrected pixel.
- Added `_two_dvr_fg_bad_width()` and tests proving the env gate is opt-in.

Usage for visual validation:
- Leave unset for current default cleanup (`1920w -> 8`).
- Set `PT_TWO_DVR_FG_BAD=0` to disable and compare against the old behavior.
- If white hair/edge lines remain, try `PT_TWO_DVR_FG_BAD=12` or `16` and check
  for hair/detail over-cleaning.
- Detailed troubleshooting notes are in
  `summary/summary_20260617_2DVR_WHITE_VERTICAL_STRIPES_CN.md`.

Late 2DVR seam update:
- `_two_dvr_rim_width()` now defaults to `max(2, round(src_w / 120))`
  (`1920w -> 16`, `3840w -> 32`), with `PT_TWO_DVR_RIM=0` as the explicit
  diagnostic/off switch.
- This does not add a new kernel. It reuses the existing `fw_hole_from_inv` rim
  branch, which is direction-gated and low-near gated.
- Debug comparison sheet:
  `debug_output/2dvr_base518_left_hair_seam_20260617/rim_times_t03_t10/rim_0_8_16_t03_t10_focus_2x.png`.

## SI progressive virtual MP4 M1 开工

### 已完成

- `pipeline/si_virtual_mp4.py` 从基础件扩展为 M1 progressive virtual remux 构造器：
  - 读取源 MP4 的 `ftyp/moov` 与视频样本 offset/size/DTS；读取混音 AAC sidecar 的音频样本表。
  - 用解析式 box writer 构造 progressive `moov`，复制源视频/音频 `stsd/stts/ctts/stss/stsz` 等稳定表，只重写 `stsc` 为 one-sample-per-chunk，并输出 `co64`。
  - 输出逻辑布局为 `memory(ftyp+moov+mdat header) + file(video samples from source) + file(audio samples from cached AAC sidecar)`，全量 `Content-Length` 精确。
  - 生成稳定 `ETag`；布局按 `(source stat, si.wav stat, SIMixParams)` 做进程内缓存，AAC sidecar 做磁盘缓存。
  - `iter_virtual_range()` 改为复用文件句柄，避免 one-sample-per-region 时反复 open 文件。
- `build_mixed_audio_sidecar()` 使用现有 `build_si_mix_filter()` 生成小 AAC MP4 sidecar；优先缓存到 `runtime_cache/si_virtual_mp4`，当前环境该目录创建有权限问题时自动 fallback 到 `debug_output/si_virtual_mp4_cache`。
- `/media_si/{name}` 已改为 M1 progressive virtual Range 服务，受 `PASSTHROUGH_SEEK_ENABLED` 门控保护：
  - 门控关闭时 404，避免旧失败传输层继续默认暴露。
  - `HEAD/GET` 返回 `video/mp4`、`Accept-Ranges: bytes`、精确 `Content-Length`、`Content-Range`、`ETag`、`DLNA.ORG_PN=HEVC_MP4_MAIN;OP=01;CI=0`。
  - GET 直接用虚拟布局切片，不再调用旧 `SIStreamService.open_stream()`，不再按 Range 现起 fragmented-MP4 ffmpeg。
- DLNA `[SI]` 条目现在默认隐藏；只有 `PASSTHROUGH_SEEK_ENABLED=1` 且 `PASSTHROUGH_SEEK_DLNA=1` 时才进入 Browse，用于 M1 灰度。协议信息改为 `HEVC_MP4_MAIN` + `CI=0`。
