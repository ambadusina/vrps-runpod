# Handover 2026-05-28

## Passthrough Multi-Concurrency

### Motivation

- Player-side `503 busy` / `409 preempted` errors are frequent on large-VRAM
  GPUs because `PT_PASSTHROUGH_MAX_CONCURRENT` defaulted to `1`.
- User asked whether the realtime server can be extended to allow 1-2
  additional concurrent GPU HTTP requests when the card has VRAM headroom, to
  significantly reduce player error returns.

### Root Blocker (Not Just a Number)

The existing `PT_PASSTHROUGH_MAX_CONCURRENT` config knob could already accept
values > 1, but raising it alone would have **broken matting output** because
`pipeline.matting.get_matter()` returned a process-wide singleton that holds
per-stream recurrent state:

- `self._rvm_rec`, `self._rvm_rec_ort`, `self._rvm_io_outputs`,
  `self._rvm_state_slots`
- shared GPU temp buffers (`_g_frame/_g_alpha/_g_out/_g_chw`)
- cached previous-frame alpha (`_cached_alpha_small`, `_cached_alpha_shape`)
- `_SceneCutDetector`, `_AlphaSmoother`

Two simultaneous streams sharing this state would have caused alpha mask
crosstalk and visible matting glitches.

### Implementation

**Matter instance pool + adaptive concurrency cap**, keeping Matter's public
surface unchanged.

#### Adaptive concurrency cap (`utils/gpu_requirements.py` + `config.py`)

- `PASSTHROUGH_MAX_CONCURRENT` default value changed to `"auto"`.
- `config.py` reads the env value and calls
  `utils.gpu_requirements.resolve_passthrough_max_concurrent()`.
- Added `detect_nvidia_total_vram_gib()` that queries
  `nvidia-smi --query-gpu=memory.total --format=csv,noheader,nounits`.
- `resolve_passthrough_max_concurrent(raw)` maps:
  - explicit integer  -> use as-is
  - VRAM >= 20 GB     -> 3
  - VRAM >= 12 GB     -> 2
  - otherwise / probe failure -> 1

#### Matter pool (`pipeline/matting.py`)

- Replaced singleton with `_pool_all` / `_pool_available` and a
  `threading.Condition` for blocking acquire.
- New API:
  - `configure_matter_pool(max_concurrent)` -> set capacity at startup.
  - `acquire_matter(blocking=True, timeout=None)` -> lazy create up to cap.
  - `release_matter(instance)` -> return + notify.
- Kept `get_matter()` for warmup and tool scripts. It always returns slot 0
  **and slot 0 stays inside `_pool_available`** so a later `acquire_matter()`
  can reuse it. This avoids the deadlock where `MAX_CONCURRENT=1` would have
  warmup permanently own slot 0 and starve real playback.
- Instances are **lazy-loaded** so cold start does not pay N x ORT/CUDA load.

#### Route wiring (`http_app/routes_media.py`)

- Added `_active_matter: dict[object, Matter]` parallel to `_active_streams`.
- Two `matter = get_matter()` call sites replaced with `acquire_matter()`.
- After acquire, the matter is bound to `slot_token` under `_active_lock` so
  preempt paths never leak.
- `_replace_active_slot(old, new)` now migrates the matter binding as well.
- `_release_active_slot(stream)` looks up the matter and calls
  `release_matter` so stream / LiveSession end-of-life auto-returns to pool.
- build_stream exception paths call `release_matter` before re-raise.

#### Startup (`main.py`)

- After GPU warmup and before the `PIPELINE:` config log, call
  `configure_matter_pool(config.PASSTHROUGH_MAX_CONCURRENT)`. The existing
  `MAX_CONCURRENT=%d` log now prints the auto-resolved value.

### Post-Review Fixes (Round 1)

Code review surfaced a deadlock in the `/passthrough` (non-live) sync branch
that the initial implementation missed. When `MAX_CONCURRENT=1` and a same-
owner range preempt fires, the old behavior was:

1. `_take_active_slot` removes old stream from `_active_streams` but only
   logs "deferred close" — the old stream is **not** closed.
2. The handler then calls `acquire_matter()` synchronously on the event loop
   thread; under pool exhaustion that becomes a `threading.Condition.wait()`
   that blocks the entire loop.
3. The old stream's generator needs the loop to make progress in order to
   reach `_release_active_slot`, so the loop deadlocks.

Fixes applied to [http_app/routes_media.py](../http_app/routes_media.py):

- `/passthrough` now calls `await _close_preempted_stream(...)` after preempt
  (parity with the live path).
- Both `/passthrough` and `/passthrough_live` build paths now invoke
  `acquire_matter` via `asyncio.to_thread` with a `PASSTHROUGH_BUSY_WAIT_SEC`
  timeout. On timeout, the slot is released via `_release_active_slot` and a
  503 with `Retry-After: 2` is returned instead of hanging.

Code review also flagged that the initial summary's claim of "Matter pool
unit tests covered" was unsubstantiated (no such tests existed). That has
been corrected, and the new `tests/test_matting_pool.py` provides the
coverage that the summary now legitimately references.

### Post-Review Fixes (Round 2)

A second review pass caught a Matter leak on the `_replace_active_slot()`
failure path. The old behavior was:

1. Caller A acquires Matter and binds it to `_active_matter[slot_token]`.
2. Caller B preempts A — `_take_active_slot` pops A's `slot_token` from
   `_active_streams` but does not touch `_active_matter`.
3. A's handler calls `await _replace_active_slot(slot_token, stream)`.
   `owner is None` so the function returned False, but it also reinserted
   the popped Matter back into `_active_matter[slot_token]`.
4. The three failure-branch call sites
   ([live slot->stream:1683](../http_app/routes_media.py),
   [stream->session:2018](../http_app/routes_media.py),
   [passthrough slot->stream:2347](../http_app/routes_media.py))
   all do `stream.close()` + `return 409` — none of them call
   `_release_active_slot(slot_token)`. The exception-path `_release_active_slot`
   only runs on raised exceptions, not on `return Response(...)`.
5. Result: the Matter sits orphaned in `_active_matter`, and after enough
   preempt collisions the pool is exhausted.

Fix applied to
[http_app/routes_media.py:693-715](../http_app/routes_media.py):
`_replace_active_slot` now releases the orphaned Matter inline (outside the
lock) when the failure branch fires, instead of re-inserting it into the
dict. Centralizing the cleanup means no future caller can repeat the same
mistake.

Added regression tests in
[tests/test_routes_media_cache.py](../tests/test_routes_media_cache.py)
`ReplaceActiveSlotMatterReleaseTests`:

- `test_failure_path_releases_orphan_matter`: pre-populate `_active_matter`
  but not `_active_streams`; call `_replace_active_slot`; assert it returns
  False, `release_matter` was called once, and dict is empty.
- `test_success_path_migrates_matter_without_release`: normal migration
  must move the Matter to the new key without invoking `release_matter`.

### Post-Review Fixes (Round 3)

Round 2's inline `release_matter()` fixed the never-released leak but
introduced a release-too-early race on the
[stream -> session:2018](../http_app/routes_media.py) path, where
`old_stream` is a live stream that has already emitted `first_live_chunk`
and whose worker thread is actively using the Matter for matting.

- After `_replace_active_slot()` immediately released the Matter, the
  `await` before the caller's `stream.close()` let the event loop schedule
  other requests.
- A waiter on `acquire_matter()` could pick up the just-released Matter
  while the old stream's worker was still running, reintroducing the
  per-Matter recurrent-state crosstalk that the pool was meant to prevent.

Fix applied to
[http_app/routes_media.py:693-735](../http_app/routes_media.py):
`_replace_active_slot()` failure path now **closes `old_stream` (via
`asyncio.to_thread`) before releasing the Matter**, when `old_stream` has
a callable `close`. Errors during close are logged and swallowed so a
misbehaving close cannot pin a Matter forever. The redundant caller-side
`await asyncio.to_thread(stream.close)` at line 2018 was removed and
replaced with a comment documenting the new contract.

Two more regression tests added to `ReplaceActiveSlotMatterReleaseTests`:

- `test_failure_path_closes_old_stream_before_releasing_matter`:
  patches `release_matter` with a recorder; the recorded event sequence
  must be `["close", "release"]`.
- `test_failure_path_release_swallows_close_exception`:
  `close()` raises; the Matter is still released to the pool (otherwise a
  faulty stream would permanently shrink the pool).

### Backwards Compatibility

- Setting `PT_PASSTHROUGH_MAX_CONCURRENT=1` (explicit) restores pre-change
  behavior exactly: single instance, single active slot, same preempt rules.
- Tool scripts using `get_matter()` (e.g. warmup, offline diagnostics) are
  unaffected because slot 0 retention rules ensure they can always acquire.

### Known Limitations / Follow-ups

- Per-Matter TensorRT engines linearly scale VRAM; the VRAM mapping leaves
  headroom but very large models may need a more conservative table.
- Default CUDA stream is still shared. Per-Matter CUDA stream is needed to
  truly parallelize GPU work; this turn intentionally scoped to fixing the
  semantic-correctness blocker first.
- NVENC session cap on legacy drivers (3) can be probed in
  `resolve_passthrough_max_concurrent` later if needed.
- `_light_match_version` is held per instance; `get_light_match().version` is
  process-global, so all instances pick up the next-frame change. No action
  needed unless behavior changes are observed.

## Seekable Realtime Passthrough Research

### User Question

The current realtime playback mode is exposed as live MPEG-TS plus coarse
chapter entries. Users prefer dragging a normal progress bar, but many target
players do not use DLNA `TimeSeekRange.dlna.org` reliably and instead issue
HTTP byte `Range` requests. User asked whether the server should estimate a
virtual total output size from video size / original bitrate / output bitrate /
original FPS / output FPS, then map incoming byte ranges back to a start time.

### Current Code Findings

- `/passthrough_live/{name}` is the current DLNA-visible realtime path. It
  accepts query `t` and `TimeSeekRange.dlna.org`, but normally advertises
  live-like behavior (`Accept-Ranges: none`, no `Content-Length`) and rejects
  most non-zero byte ranges for non-managed live clients.
- `dlna/content_directory.py` deliberately hides legacy pseudo-VOD
  `/passthrough/{name}` from DLNA and exposes a chapter container instead.
  The inline reason is that several clients issue aggressive probe/seek
  requests that are a poor fit for on-demand generated media.
- `/passthrough/{name}` already implements the user's proposed model:
  `_estimated_passthrough_size()` declares a virtual total size,
  `_seek_from_byte_range()` maps `Range.start / total * duration` to seconds,
  and responses include `Accept-Ranges: bytes`, `Content-Range`,
  `Content-Length`, plus optional padding to the declared length.
- `utils/bitrate_estimator.py` already stores EWMA bitrate estimates keyed by
  source stat data and encoder settings, with UTF-8-sig JSON reading for BOM
  tolerance.

### External Research Notes

- HTTP byte range semantics are byte offsets inside one selected
  representation. `Content-Range` and `Accept-Ranges` should describe that
  representation consistently; a synthetic generated stream is therefore a
  compatibility layer, not a fully standards-pure static file.
- Browser/native players generally compute byte ranges from container seek
  metadata or extractor seek maps. Android Media3/ExoPlayer derives the HTTP
  `Range` header from an internal data position and length; its docs also
  note that some formats are only approximately seekable using constant
  bitrate assumptions.
- DLNA-style `X-AvailableSeekRange` / `TimeSeekRange.dlna.org` is the cleaner
  time-seek path, but target-player support is not dependable enough to rely
  on it alone.

### Recommended Direction

- Do not mutate `/passthrough_live` into a general byte-seek VOD endpoint.
  Keep live chapters as fallback and revive `/passthrough` or introduce a
  sibling `/passthrough_seek` endpoint behind a feature flag.
- Treat the endpoint as a byte-addressed **virtual CBR representation**:
  stable virtual `Content-Length`, `Accept-Ranges: bytes`, deterministic
  byte->time mapping, and generated output starting from a clean keyframe.
- Prefer MPEG-TS for this compatibility layer. It can restart from PAT/PMT +
  IDR after any seek more naturally than MP4, whose seek behavior depends on
  truthful `moov` sample tables and byte offsets.
- Base the declared size primarily on output bitrate + duration + audio bitrate
  + container overhead + learned EWMA. Source bitrate/FPS should only inform
  adaptive output caps and initial estimates; the player is seeking the
  generated output byte space, not the source file byte space.
- Improve mapping before exposing broadly:
  reserve a small virtual header/probe prefix, map `(range_start -
  header_bytes) / (total - header_bytes)`, clamp, then snap backward to a GOP
  or IDR boundary. Add response headers/logs for mapped time and estimate
  source for diagnostics.
- Real compatibility requires a device matrix. Capture actual request patterns
  for 4XVR, Skybox, Moon, DeoVR, HereSphere, nPlayer, VLC/mpv: initial HEAD,
  `bytes=0-`, small bounded head probes, tail probes, and mid-stream open
  ranges.

### Risk Summary

- A pure byte->time mapping violates the assumption that every byte range is
  part of the same immutable encoded file. Some players will stitch/cache
  ranges or parse indexes and fail if returned bytes do not match the declared
  representation.
- MP4 is especially risky unless the generated file or fragment index is real.
  Accurate MP4 byte seek likely requires offline/nearline generation of the
  output file or a segment index with truthful offsets.
- The route should be opt-in per player/profile until real-device validation
  confirms which players tolerate the pseudo-VOD behavior.

### Evaluation of `summary_20260528_SEEKABLE_PASSTHROUGH_RESEARCH_CN.md`

Overall verdict: the proposed direction is sound. It correctly separates
seekable pseudo-VOD from `/passthrough_live`, reuses the hidden `/passthrough`
assets, chooses MPEG-TS as the first target container, and treats byte seek as
a virtual-CBR compatibility layer rather than true file byte seek.

Important corrections before implementation:

- DLNA Browse-time player profile is not reliably known. The existing
  `utils.player_compat` classification is request-time and UA-based. Therefore
  `PT_PASSTHROUGH_SEEK_DLNA=profile_list` cannot be the only exposure model.
  Prefer an explicit global/dual-listing switch first, then route-time policy
  can reject or fall back per UA.
- The plan applies `LiveSession` key quantization to seek mode, but
  `/passthrough` does not currently use `LiveSession`. Reusing one producer for
  multiple mid-range subscribers is not a small key change; it needs a separate
  seek-session/cache design. First milestone should rely on same-client
  preempt + probe cache + debounce, not shared producers.
- `header_reserved` must be treated carefully. If it is only a virtual no-seek
  zone, ranges inside it must still return bytes that are consistent with byte
  offsets already served from `0-`, preferably from real cached TS prefix
  bytes. A fabricated prefix can poison strict parsers.
- Current small-probe fallback synthesizes an MP4 `ftyp` prefix. In MPEG-TS
  seek mode this must be removed or replaced with real producer-generated TS
  prefix bytes. Handcrafted PAT/PMT is risky; let FFmpeg/PyNv mux produce it.
- The statement "input `-ss` already snaps to keyframe" is only true for the
  FFmpeg fallback path in a loose sense. The PyNv path maps start time to frame
  index and forces output IDR on the first encoded frame. GOP snapping is mainly
  for cache/preempt stability and predictable startup, not decoder correctness.
- `Content-Length` stability needs a session policy. EWMA may update the
  estimate after a complete stream; that is good long-term but can surprise
  clients that cache metadata for the same object. Avoid changing declared size
  during active sessions, and expose seek items behind a DIDL version/config
  switch while validating.
- The risk note suggesting "declare size down by 5-10%" is not obviously safe.
  Under-declaring can make the player unable to reach the true end. The current
  estimator intentionally has an upward safety multiplier plus optional
  padding; keep that until device data proves otherwise.

Recommended implementation order:

1. Keep the summary's data-gathering step.
2. Add a hidden/explicit seek endpoint or query mode first, forced to MPEG-TS
   without changing global `PASSTHROUGH_CONTAINER`.
3. Implement a tested byte-range-to-time helper with no-seek prefix handling
   and GOP snapping.
4. Fix MPEG-TS probe behavior to use cached real prefix or controlled 503,
   never MP4 synthetic bytes.
5. Add same-client same-file preempt policy for seek requests.
6. Only then expose DLNA items via a conservative config switch, ideally
   alongside live chapters during validation.

- Remaining cleanup recommended before treating the summary as final:

## DLNA Player Compatibility Refactor Notes

Context: user asked for a design pass on DLNA/player compatibility handling.
The current code has grown by stacking User-Agent branches around
`/passthrough_live`, mostly to protect small-VRAM users from probe/thumbnail
requests stealing the only realtime passthrough slot.

Relevant current code:

- `http_app/routes_media.py` owns raw `/media`, `/thumb`, `/passthrough`, and
  `/passthrough_live` requests.
- `_live_response_profile()` maps User-Agent to `nplayer`, `avpro`, `libmpv`,
  `4xvr`, `vlc`, `lavf`, or fallback profile.
- `/passthrough_live` currently embeds policy decisions inline:
  managed session profiles, non-start Range rejection, Lavf rejection, nPlayer
  duplicate-startup debounce, VLC preroll/pseudo-VOD, and active-slot preempt.
- `_take_active_slot()` / `_can_preempt_owner()` implement the resource guard.
- `LIVE_REQUEST_HEADER_DUMP` only dumps live headers to
  `debug_output/live_requests`; it does not capture raw media, thumbnails, CDS
  Browse, or request ordering.
- `/media` is ordinary Range file serving and does not use GPU, but its request
  pattern is important for distinguishing player thumbnails, preview scrubbing,
  tail probes, and real playback.
- `/control/cds` does not currently log/store request context beyond normal
  route handling.

Recommended direction:

1. Split the current inline behavior into three layers:
   request observation -> player/profile resolution -> request-intent policy.
2. Add a bounded request history ring buffer that records all DLNA-relevant
   requests, not just `/passthrough_live`: CDS Browse, `/thumb`, `/media`,
   `/passthrough_live`, `/passthrough`, subtitles, HEAD/GET, Range,
   TimeSeekRange, UA, client host, status, bytes/chunks where available, and
   timing.
3. Persist periodic JSONL snapshots to `debug_output/request_history/` so real
   device traces can be checked without Wireshark.
4. Treat User-Agent only as one signal. Use recent request sequence, method,
   path kind, Range shape, TimeSeekRange, transferMode/getcontentFeatures,
   repeated same URL, and active playback state to classify intent:
   `playback_primary`, `startup_probe`, `duplicate_startup`, `thumbnail_probe`,
   `tail_probe`, `side_probe`, `browse`, `subtitle`, `unknown`.
5. Introduce profile policies as data/config instead of hard-coded branches:
   profile match rules, managed-session usage, range policy, side-request
   rejection, preroll size/timeout, pseudo-VOD response shape, audio override,
   preempt rules, and cache key behavior.
6. Keep a global mode such as `PT_DLNA_PLAYER_PROFILE=auto|vlc|libmpv|nplayer|
   4xvr|avpro|deovr|strict_live` so users can force a known path when auto
   detection misclassifies a client.
7. Preserve the key invariant for small VRAM: any request classified as probe,
   thumbnail, tail, or side request must be answered from cache/synthetic data
   or rejected before acquiring Matter/PyNv/NVENC; only primary playback may
   acquire or preempt active passthrough resources.

Suggested first implementation phase should be observation-only:

### Seekable Passthrough fMP4 Prefix-Crossing Range Fix

User tested true fMP4 seek with nPlayer. The first no-Range request returned
200 OK with `Content-Type: video/mp4`, but nPlayer immediately requested:

`Range: bytes=589824-`

The request is inside the reserved seek prefix region
(`589824 < PT_PASSTHROUGH_SEEK_HEADER_BYTES`, default 2 MiB) while extending to
EOF. Logs confirmed the old behavior:

`passthrough_seek[21] stream header-crossing range from start with skip=589824`

That path started a fresh fMP4 producer at `t=0` and discarded 589824 bytes
before returning body data. For MP4/fMP4 this is too fragile: a client is
asking for bytes from the same advertised representation, so the reserved
prefix must be byte-identical to the original startup response.

Fix applied:

- Added `_is_header_crossing_range()` and `_seek_prefix_splice_or_retry()`.
- For any non-zero Range crossing the reserved prefix boundary, seek now waits
  briefly for a complete cached muxer prefix.
- On cache hit, the response first yields the real cached prefix slice
  `[range.start, header_limit)`, then starts a new producer from `t=0` and
  discards exactly `header_limit` bytes before appending the rest.
- On cache miss, the route returns `503 + Retry-After: 1` before acquiring a
  Matter or starting a new encoder stream, instead of preempting the startup
  stream and producing inconsistent prefix bytes.
- Startup seek streams now publish the prefix cache at 64 KiB steps up to the
  reserved limit, plus the existing final write. With the default 2 MiB prefix
  this is bounded and avoids the previous 16 MiB per-chunk O(N^2) behavior,
  while making early nPlayer ranges such as `589824-` visible to later
  requests.
- Removed the obsolete direct header-crossing skip helper so this fragile path
  is not accidentally reintroduced.

Legacy `/passthrough_live` behavior was not changed.
