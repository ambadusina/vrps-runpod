## Throughput optimization (SMALL, inverse_warp, flat3d)

Test file `videos/test_2d_4k.mp4` is **actually 1920×1080** (HEVC 60fps), not 4K.

| proc res | before | after |
|---|---|---|
| 1920 (full 1080p) | 1.5 fps | **27 fps** |
| 1600 | — | 35.6 fps |
| 1280 | — | **47.6 fps** |

~18× faster. **40 fps lands at ~1450px.** Wins:

- **Depth 38 → ~5 ms**: TensorRT fp16 EP (fuses ViT, kills the 53 CUDA memcpy
  nodes) + folded preprocessing (normalize on GPU).
- **Render 89 → ~16 ms**: low-res disparity normalize then upscale (was 42 ms of
  full-frame `np.percentile`+mask), preallocated grids/buffers, cv2.remap /
  GaussianBlur / dilate, in-place `np.subtract|add`.
- **Pipeline**: 3-stage threads so TRT depth overlaps cv2 render; writer streams
  the ndarray via buffer protocol (no `tobytes`).

## Known limits / next

- **Full 1080p caps ~27 fps** (render = 2× full-res cv2.remap on CPU + GIL). True
  40 fps full-res / real 4K needs a GPU-resident render, which is **blocked here**:
  GPU is **RTX 5060 Ti sm_120 (Blackwell)** — CuPy 14 has no sm_120 kernels
  (`NO_BINARY_FOR_GPU`), torch absent, `cv2.cuda` 0 devices. Only ORT does GPU.
- Practical 40 fps+ path **today**: process at ≤1280 (the `fast` quality tier).
- To unlock full-res/4K high fps: install torch cu128 and rewrite the warp on GPU
  tensors (or wait for a CuPy sm_120 wheel).
- soft_shift is the slow quality mode (forward-warp scatter is numpy/CPU-bound).
- ORT warning `ScatterND with reduction=='none'` from the depth graph's
  `x[:,:,0]=cam_token` slice-assign — benign (numerics validated).

## CuPy sm_120 (Blackwell) NVRTC fix — `pyproject.toml`

Project uses CuPy heavily across the realtime/offline pipeline (pynv_stream,
matting, alpha_guided_filter, pynv_io, matanyone2_engine…). On the RTX 5060 Ti
(**sm_120**) CuPy's NVRTC kernels hit `CUDA_ERROR_NO_BINARY_FOR_GPU` with system
CUDA 12.6 (or fall back to the PTX→driver-JIT cold-start that the existing
`CUPY_COMPILE_WITH_PTX=1` strategy relies on).

- **Fix (one dep)**: `nvidia-cuda-nvrtc-cu12>=12.8,<12.9` added to `pyproject.toml`.
  Gives CuPy a sm_120-native NVRTC → RawKernel/ElementwiseKernel compile straight
  to sm_120 cubins, **no driver-JIT cold-start**. Reproducible via `uv sync`,
  independent of system CUDA. No CuPy version change.
- **Why not bump/downgrade CuPy**: CuPy 14.0.1 has a vendored libcudacxx
  `__nv_fp8_e8m0` header bug that breaks cccl/thrust kernels (percentile, sort,
  `cupyx.scipy.ndimage`) under NVRTC 12.8 **and** 12.9. The pipeline uses **none**
  of those (verified by grep), so CuPy 14 is fine; 12.8 is the safe NVRTC floor.
  CuPy 13.6 lacks the bug but its NVRTC loader ignored the pip wheel and used
  system 12.6 → not viable without a system CUDA upgrade.
- **Verified**: `uv sync` clean (removed cupy13-experiment orphan `fastrlock`);
  CuPy sm_120 direct-cubin OK (RawKernel, no PTX env); DA3 depth still on
  TensorRT; offline 2D→VR end-to-end OK (1280 → 42 fps).
- **⚠️ Frozen build TODO (not done here)**: `build_exe.bat` copies
  `nvrtc-builtins*.dll` from *system* CUDA and sets `CUPY_COMPILE_WITH_PTX=1`. To
  ship sm_120-native to 50-series users, bundle the pip NVRTC from
  `site-packages/nvidia/cuda_nvrtc/bin` and update `packaging/hooks/hook-cupy.py`
  + `runtime_hook_cuda_dlls.py`.

## DA3 TRT vs Offline RVM/MatAnyone2 TRT manifest warmup analysis

Compared `ui/pages/two_dvr_page.py` DA3 TensorRT flow with
`ui/pages/offline_page.py` RVM/MatAnyone2 TensorRT cache flow.

- DA3 UI only passes `--provider trt|cuda` into `offline/two_dvr.py`. The actual
  runtime is `offline/da3_depth.py`: ORT provider chain
  `TensorrtExecutionProvider -> CUDAExecutionProvider -> CPUExecutionProvider`,
  with `trt_fp16_enable=True`, ORT engine cache under
  `runtime_cache/da3_trt/<small|base>`, and no manifest/fingerprint validation.
- DA3 configure dialog (`ui/widgets/da3_trt_dialog.py`) treats cache readiness as
  "does any `.engine` exist per variant". Build is `offline/two_dvr.py build-trt`
  and runs one dummy 720p frame through each DA3 variant to force ORT TRT engine
  creation.
- Offline RVM/MatAnyone2 UI uses `utils/trt_manifest.py`: manifest includes GPU,
  driver, TRT/ORT/CUDA versions, model hash, precision/settings, and required
  engine artifacts. UI only enables child TRT env when `cache_status(...) ==
  "ready"`.
- Offline RVM warmup builds static-shape ONNX variants for all offline precision
  tiers: `(1024,b1,0.5)`, `(1024,b2,0.5)`, `(2048,b1,0.25)`,
  `(2048,b2,0.25)`, `(2048,b1,0.5)`, `(2048,b2,0.5)`. Runtime then loads static
  TRT sessions for matching shapes; main dynamic RVM session is kept off TRT.
- MatAnyone2 TRT is narrower: only `matanyone2_step_update.onnx` uses TRT, with
  separate cache subdirectories for 512 and 1024 bs1. Other MatAnyone2 graphs
  stay CUDA. Warmup runs isolated subprocesses per model key to avoid ORT/TRT
  process-global cache reuse issues.

Conclusion: DA3's path is lighter and faster to build/run for its workload, but
less robust for cache validity. Offline manifest warmup is heavier but safer and
necessary for RVM/MatAnyone2 because shape/state/cache correctness is more
fragile. Efficiency is workload-specific: DA3 TRT is most efficient as a simple
per-frame ViT depth accelerator; RVM static TRT is efficient only on supported
precision shapes; MatAnyone2 TRT accelerates only the step-update hot path and
does not make the whole model TRT-resident.

## DA3 folded-preprocess TensorRT UINT8 parser log handling

Observed UI log during 2D->3D/VR DA3 TRT startup:

- ORT TensorRT EP logs `TensorRT does not support UINT8 types for intermediate
  tensors` while parsing the first `Transpose` node.
- Conversion continues with `depth=TensorrtExecutionProvider` and normal high
  FPS, so this is not a runtime failure.

Cause:

- The folded-preprocess DA3 ONNX accepts `uint8` NHWC image input.
- TensorRT tries to parse the leading UINT8 preprocessing/transpose subgraph,
  logs an error for that unsupported node, then ONNX Runtime partitions the
  unsupported preprocessing to CUDA and keeps the supported DA3 depth trunk on
  TensorRT.

Fix:

- In `offline/da3_depth.py`, DA3 sessions created with `provider="trt"` now set
  `SessionOptions.log_severity_level = 4`.
- Scope is intentionally narrow: only DA3 TRT session construction is quieted.
  The app still logs explicit DA3 auto-build failures from `offline/two_dvr.py`.

Verification:

- `python -m py_compile offline/da3_depth.py offline/two_dvr.py
  ui/pages/two_dvr_page.py` passed with `PYTHONPYCACHEPREFIX` pointed at
  `%TEMP%\ptmediaserver_pycache_check`.
