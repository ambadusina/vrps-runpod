# HANDOVER 2026-05-21

## DLNA and Offline VR Naming Update

This section records the final post-merge state for the filename work. Earlier intermediate assumptions about square 1:1 sources and space-separated markers were corrected during the same development thread.

- Centralized VR filename logic in `utils/vr_naming.py`.
  - DLNA naming, offline default output naming, and filename marker checks now share this module.
  - The marker set covers player-style suffixes such as `LR/RL/LRF/RLF/SBS/SBSF/HSBS/TB/BT/TBF/BTF/OU/OUF/HOU/2D/3D/3DH/3DV/F180/180F/FISHEYE/FISHEYE190/RF52/MKX200/MKX22/VRCA220/EAC360/360EAC/ALPHA`.
  - Valid marker separators are only underscore and hyphen: `_` and `-`.
  - Spaces and ` - ` are intentionally not valid separators. Examples: `scene F180` and `scene - F180` do not count as VR/player markers.
  - Dot is not a marker separator. Example: `scene.F180` does not count as `_F180`.
- Extension parsing in `utils.vr_naming._as_stem()` is explicit:
  - It takes only the substring after the final dot as the extension candidate.
  - Only known video extensions are stripped.
  - Dots inside source names are preserved. Regression case: `xxxx.com@atvr00067_1_8k.mp4` becomes `xxxx.com@atvr00067_1_8k_SBS_180`, not `xxxx_SBS_180`.
- Source-video naming rules:
  - Plain 2D videos are width:height < 2:1 and keep their original DLNA title.
  - 2:1 half-equirectangular / VR180 sources without known player markers are virtually displayed as `source_SBS_180`.
  - Existing VR/player markers prevent automatic `_SBS_180` injection.
- DLNA naming changes:
  - Raw media title uses `source_display_stem()`.
  - Green live virtual title is based on the displayed source stem, e.g. `movie_SBS_180_passthrough_live`.
  - Alpha live virtual title is based on the original source stem and uses SBS + F180 markers, e.g. `movie_SBS_F180_alpha_live`.
  - `FISHEYE` in alpha source names is rewritten to `F180` before appending the alpha suffix.
  - `dlna/content_directory.py` DIDL schema version was bumped from 4 to 5 to avoid stale client/catalog cache.
- Offline naming changes:
  - `offline/convert.py`, `tools/offline_passthrough.py`, and `tools/offline_alpha_passthrough.py` call `utils.vr_naming`.
  - Green offline defaults:
    - 2:1 unmarked source: `source_SBS_180_passthrough.mp4`
    - normal 2D source: `source_passthrough.mp4`
  - Alpha offline defaults: `source_SBS_F180_alpha.mp4`.
  - `utils/offline_outputs.py` recognizes the new suffixes while keeping legacy suffixes.
- Tests added/updated:
  - `tests/test_vr_naming.py`
  - `tests/test_content_directory_modes.py`
  - `tests/test_offline_convert.py`
  - `tests/test_offline_outputs.py`

## 2026-05-22 server.log playback stutter analysis

- Reviewed `debug_output/server.log` after user reported that some files still stutter.
- TensorRT/RVM matting does not appear to be the cause in this log:
  - static TRT path is active: `static_trt=True`, `iobinding=True`;
  - realtime `ort_run` is consistently around `5.4-6.0ms` for `(2,3,1024,1024)`;
  - no `run_with_iobinding()` hang or 30s first-chunk timeout appears.
- Observed real stutter signal:
  - several sessions log `mux stdin write slow`;
  - this means encoded bitstream writes into the ffmpeg mux stdin blocked for about `1.0-1.9s`;
  - affected examples include:
    - `twojav.com@kiwvr00832_1_8k.mp4`;
    - `hhd800.com@mdvr00297.part4_8K.mp4`;
    - `sivr00468_2_8k.mp4`;
    - `hhd800.com@sivr00390.part3_8K.mp4`.
- Current interpretation:
  - the bottleneck is downstream mux/client/audio interleaving backpressure, not matting inference;
  - the current `pipe_ts` two-ffmpeg mux path can block video writes when the final MPEG-TS muxer is waiting on downstream output or audio interleaving.
- Also fixed misleading cleanup noise:
  - audio cache interruption previously terminated the ffmpeg process and closed pipes before `communicate()`;
  - Python's internal subprocess reader thread could then print `ValueError: I/O operation on closed file`;
  - changed audio cache stop calls to `close_pipes=False` so `communicate()` can drain/close pipes normally.
- Verification:
  - `.\.venv\Scripts\python.exe -m compileall pipeline\pynv_stream.py`
