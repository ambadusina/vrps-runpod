## NVDS Model Shape And Placement

- `reference/NVDS/full_model.py` defines `NVDS(nn.Module)`.
- Forward input is conceptually `[B, 4, 4, H, W]`:
  - batch,
  - 4-frame temporal window,
  - 4 channels = RGB + initial depth/disparity,
  - inference height/width.
- Forward output is `[B, 1, H, W]`, a stabilized target-frame depth/disparity.
- The model uses `mit_b5()` backbone and changes the first conv from 3 to 4
  channels.
- It is a postprocessor, not a DA3 replacement:
  `RGB frame + DA3 depth/near sequence -> stabilized depth/near -> stereo renderer`.
- Best first integration point:
  - after DA3 depth inference in `offline/two_dvr.py` / `offline/two_dvr_pynv.py`,
  - before `renderer.render(...)` or `renderer.prepare_near_gpu(...)`.
- For the CPU/ffmpeg path, feed `renderer.render(frame, stable_depth_or_near)`.
- For the PyNv path, add an ONNX stabilizer step before
  `renderer.prepare_near_gpu(...)`; this is feasible but more invasive because
  the path currently keeps RGB/depth mostly on GPU and only round-trips DA3
  input/output through ORT.

## Export Risks

- Export should be fixed-resolution first. Dynamic H/W is high risk due to
  Python shape arithmetic, padding branches, window partitioning, and many
  `view/reshape` operations in `stabilization_attention.py`.
- The largest ONNX risk is in-place slice updates such as:
  `attn[:, :, ...] = attn[:, :, ...] + bias`.
  Newer PyTorch exporters may lower some of these to Scatter/Where-style ONNX
  ops, but PyTorch 1.9 exporter is likely weaker.
- Recommended first export resolution:
  - proof: `384x384`,
  - real 16:9 offline test: `672x384` (closest multiple-of-32 width for height
    384),
  - optional higher quality: `896x512` or other fixed aspect models only after
    memory/perf is measured.
- README states the stabilizer alone uses about 5GB GPU memory at `896x384` on
  RTX-A6000. This is not a realtime candidate in this project. Treat it as
  offline-only.
- The checkpoint is loaded in the official script into `torch.nn.DataParallel`,
  so export code may need to strip `module.` prefixes or load into DataParallel
  first and export `model.module`.

## Input Normalization Risk

- Official inference normalizes RGB with ImageNet mean/std.
- Official inference normalizes the initial depth/disparity per frame to `[0,1]`
  before concatenating it with RGB.
- Current DA3 output is distance-like depth, and the renderer converts it to
  normalized near/disparity by reciprocal + percentile band.
- Because NVDS was trained on DPT/MiDaS-style relative inverse depth/disparity,
  the safer first experiment is to feed NVDS a normalized near/disparity map,
  not raw DA3 distance depth.
- If output is treated as near/disparity, renderer integration should bypass the
  reciprocal step or add a dedicated `render_from_near(...)` path. If output is
  converted back into distance-like depth, extra inversion and rescaling are
  needed. This needs A/B validation.

## nunif iw3 Depth Stabilization Reference

- User asked what depth stabilizer `reference\nunif` uses for 2D-to-3D.
- Relevant tool is `reference\nunif\iw3`.
- It does not use NVDS as a depth stabilizer.
- Main video-depth stabilization is `Flicker Reduction` / `--ema-normalize`:
  - Implemented by `reference\nunif\iw3\depth_scaler.py:EMAMinMaxScaler`.
  - It tracks per-frame/minibatch depth min/max, optionally with a lookahead
    window, then smooths min/max by exponential moving average before
    min-max normalization.
  - CLI params are `--ema-normalize`, `--ema-decay` default `0.75`, and
    `--ema-buffer` default `30`.
  - README recommends decay around `0.75` to `0.99`; buffer 30 means roughly
    1 second lookahead at 30 FPS.
- Scene cut handling:
  - `--scene-detect` uses TransNetV2 first.
  - At scene boundaries, nunif resets VideoDepthAnything state and Flicker
    Reduction / Min-Max normalization state.
- VDA path:
  - `VideoDepthAnything` / `VideoDepthAnythingStreaming` are video depth models
    with their own temporal model state.
  - nunif still recommends combining them with `--ema-normalize` and
    `--scene-detect`.
- `row_flow_v3` is the default stereo generation method. It is for learned
  backward warping / stereo synthesis, not depth temporal stabilization.
- Real-time/desktop path in `reference\nunif\iw3\desktop` enables Flicker
  Reduction by default but forces `buffer_size=1`, avoiding lookahead latency.

## 2DVR Default Stabilizer vs nunif Reference

- Current built-in 2DVR stabilizer already overlaps with nunif's main idea:
  - `offline/two_dvr_render.py:_normalize_near(...)` converts DA3 distance to
    inverse-depth/near and uses the 5/95 percentile band.
  - `TemporalDepthStabilizer.normalization_band(...)` EMA-smooths that band.
  - Our alpha maps to nunif decay as `decay = 1 - alpha`; current defaults are
    norm alpha `0.10` (decay `0.90`) and depth/base alpha `0.20` (decay `0.80`),
    both inside nunif's suggested `0.75..0.99` decay range.
- Current built-in stabilizer goes beyond nunif:
  - base/detail split on normalized near/disparity;
  - only smooths low-frequency base, always reinjects current detail;
  - optional global translation motion compensation;
  - affine scale/bias match;
  - evidence gate to let moving/disoccluded regions follow current frame.
- Existing offline lookahead:
  - `SymmetricBaseWindow` exists in CPU code, and `GpuSymmetricWindow` exists in
    GPU code.
  - PyNv path uses `GpuSymmetricWindow` when `--temporal-window > 0` and disables
    causal temporal-depth smoothing there.
  - The ordinary ffmpeg/CPU pipeline currently does not consume
    `--temporal-window`, so offline lookahead is not consistently applied across
    paths.
- Main borrowable nunif ideas:
  - Add lookahead/symmetric smoothing for the normalization band itself, not only
    the base/near map. This is closest to nunif `--ema-buffer`.
  - Add scene-cut detection/reset for 2DVR. The repo already has
    `utils.scene_detection.SceneCutDetector`, but 2DVR does not currently use it.
    Reset should clear renderer temporal state, NVDS state when selected, and any
    symmetric window buffer at cut boundaries.
  - Avoid using NVDS as the default path unless performance is solved; nunif's
    stabilizer is algorithmic EMA/range reset, not a separate heavy model.

## NVDS Freeze Root Cause And VRAM Fix (commit 28499d5)

- User reported NVDS run getting progressively slower (`1.9 -> 2.1 -> 1.2 fps`)
  then freezing. Reproduced with a live `nvidia-smi` memory trace.
- Root cause is NOT "NVDS too slow": it is VRAM exhaustion. DA3 (TRT) + NVDS
  (CUDA EP) + CuPy renderer share the GPU; real working set is ~14.9 GB / 16 GB.
  ORT's default arena uses `kNextPowerOfTwo` and never frees, so combined usage
  creeps past physical VRAM and Windows WDDM spills to shared system RAM -> the
  progressive slowdown ending in an effective freeze. Serial mode was worse
  (0.1 fps) because it relieved no memory pressure.
- Fix in `offline/nvds_stabilizer.py`: give the NVDS `CUDAExecutionProvider` a
  `gpu_mem_limit` auto-derived as `total VRAM - _NVDS_VRAM_RESERVE_MB` (default
  reserve 4096 MB). Keep ORT's fast default arena/cudnn kernels -- a first
  attempt using `kSameAsRequested` + `cudnn_conv_algo_search=DEFAULT` also stopped
  the freeze but cut fps from 2.8 to 0.4, so it was reverted.
- Env overrides: `PTMS_NVDS_GPU_MEM_LIMIT_MB`, `PTMS_NVDS_VRAM_RESERVE_MB`.
- Measured on the 16 GB card: monolith 672x384 went from `1.2 fps -> freeze` to a
  stable `2.8 fps`, VRAM flat at ~12.6 GB.

## NVDS Backbone/Head Split (commit 02fe09d)

- `examples/export_nvds_onnx.py --split` exports two graphs:
  - `NVDS_Backbone_{w}x{h}.onnx`: single RGBD frame `[1,4,H,W]` -> 4 feature maps.
  - `NVDS_Head_{w}x{h}.onnx`: 4 window frames' features (stacked on batch) +
    last-frame RGB `[1,3,H,W]` -> stabilized depth `[1,1,H,W]`.
- `offline/nvds_stabilizer.py` auto-detects the split graphs, runs the backbone
  once per frame, caches the last 4 feature tuples, runs the head per frame.
  Falls back to the monolith when split files are absent.
- Numerically identical to the monolith (backbone has no cross-frame ops, so its
  features are batch-independent): `max_abs_diff ~= 3.5e-05`; the export script
  verifies split-vs-monolith in one run.
- IMPORTANT finding that corrected the earlier assumption: the backbone is NOT
  the bottleneck. Measured at 672x384 (CUDA EP, 40-frame mean):
  - backbone (1 frame) = 26 ms, head (focal cross-attention) = 208 ms.
  - monolith NVDS = 293 ms, split NVDS = 258 ms -> only 1.13x.
  The focal-attention head dominates and cannot be de-duplicated (it is the
  per-output-frame cross-frame fusion). Microbench tool:
  `tools/nvds_split_bench.py [split|mono]`.
