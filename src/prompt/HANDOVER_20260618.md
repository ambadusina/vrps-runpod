## SI Progressive Cache Stall Fix

### Finding

- User confirmed the cold first-play stall and folder Browse freeze came from
  the SI progressive cache path:
  - `_si_item_from_video()` queued prewarm for every `[SI]` item during Browse.
  - The old prewarmer spawned one daemon thread per item and serialized them
    behind `Semaphore(1)`, so a folder with many SI videos created many blocked
    threads and an unbounded amount of pending cold work.
  - Foreground `/media_si` layout builds did not share that prewarm semaphore,
    so playback could still run a second heavy ffmpeg/layout build beside a
    background prewarm.
  - `read_media_sample_table(video, "video")` used PyAV demux, causing a full
    source-file packet scan even though the needed sample table data already
    lives in `moov`.

### Change

- `pipeline/si_virtual_mp4.py`
  - Added pure `moov` parsing for video sample tables:
    - `stco/co64` + `stsc` + `stsz` expand sample offsets and sizes.
    - `stts` + `ctts` compute DTS/PTS.
    - `stss` marks keyframes.
  - `read_media_sample_table(..., "video")` now uses the `moov` parser first and
    falls back to PyAV only when parsing fails. Audio sidecar sample tables still
    use PyAV because the sidecar is small.
  - Replaced one-thread-per-prewarm with a single background worker and bounded
    priority queue.
  - Added a global cold-build slot shared by background prewarm and foreground
    playback. Background prewarm runs as low priority and can be cancelled when
    a foreground playback build arrives.
  - `build_mixed_audio_sidecar()` now uses cancellable `Popen` instead of
    blocking `subprocess.run()`. Low-priority ffmpeg processes use
    `BELOW_NORMAL_PRIORITY_CLASS` on Windows when available.
- `config.py`
  - Added `SI_BROWSE_PREWARM_LIMIT`, default `1`.
  - Added `SI_PREWARM_QUEUE_MAX`, default `2`.
- `dlna/content_directory.py`
  - Browse still exposes all `[SI]` items, but only the first
    `SI_BROWSE_PREWARM_LIMIT` SI items in a directory listing attempt background
    prewarm.
  - Later SI items are visible but do not enqueue prewarm work.
- Tests cover:
  - synthetic MP4 `moov` sample-table expansion without media demux.
  - bounded/deduped prewarm queue.
  - DLNA SI item generation with prewarm skipped.
  - new config defaults.

## SI Source Audio Cache Decoupling

### Goal

- Implement the user's step 2: decouple source-audio extraction from SI mix
  parameters and minimize first-play blocking time.
- Avoid `ffmpeg -i 8GB.mp4` for every mix-parameter change.

### Change

- `pipeline/si_virtual_mp4.py`
  - Added `build_source_audio_sidecar(video)`:
    - keyed only by source video stat and track index.
    - parses source audio sample table from `moov`.
    - writes a compact `*.source-audio.mp4` containing only source audio samples.
    - preserves the source audio trak timing/description/edit boxes and rewrites
      only `stsc/co64` for the compact cache file.
  - `read_media_sample_table()` now uses the `moov` parser for audio too, with

  - `build_mixed_audio_sidecar()` now uses the source-audio sidecar as ffmpeg
    input when available.
  - First cold mix path is optimized further:
    - when source-audio sidecar is missing, ffmpeg receives `pipe:0`;
    - Python generates the audio-only MP4 once and tees it to both ffmpeg stdin
      and the `*.source-audio.mp4` cache;
    - source-audio extraction and AAC mixing therefore overlap instead of
      running serially.
  - If the source-audio cache/pipe path fails, the code logs a warning and falls
    back to the previous full source-video ffmpeg input path for compatibility.
- `tests/test_si_virtual_mp4.py`
  - covers audio-only source sidecar construction and payload/sample table.
  - covers mixed sidecar using an existing cached source-audio input.
  - covers first-cold mixed sidecar using `pipe:0` and creating the source-audio

## SI Sidecar Build Acceleration Plan Follow-up

### Goal

- Implement `summary/summary_20260618_SI_SIDECAR_BUILD_ACCELERATION_PLAN_CN.md`.
- Shorten first `/media_si` playback preparation while preserving audio timing
  and avoiding segmented AAC seam artifacts.

### Change

- `pipeline/si_virtual_mp4.py`
  - Added sequential source-audio extraction:
    - `_copy_file_ranges_sequential_to_outputs()` scans the source file forward
      with a large buffer instead of issuing one cold seek per audio sample.
    - Non-monotonic sample offsets now fall back to the old run-copy path so
      decode order is not changed for unusual files.
  - Added segmented native-AAC implementation:
    - `_plan_mix_segments()`, `_encode_mix_segment()`,
      `_stitch_aac_segments()`, and audio `stts/stsz/co64/edts` rewrite helpers.
    - Stitching preserves the first segment priming when used and normalizes
      segment-local boundary `stts` trim artifacts.
  - Added encoder/implementation cache variant to `_cache_digest()` so sidecars
    are not reused across encoder or segmented/single implementation changes.
  - Default mix path now selects continuous `aac_mf` on Windows when available;
    native segmented AAC remains behind explicit `PT_SI_MIX_SEGMENTED_AAC=1`.
- `config.py`
  - Added `SI_AUDIO_EXTRACT_MODE=sequential`.
  - Added `SI_MIX_PARALLEL_MAX`, `SI_MIX_SEGMENT_WARMUP_MS`.
  - Added `SI_MIX_ENCODER=auto` and `SI_MIX_SEGMENTED_AAC=0`.
- `tests/test_si_virtual_mp4.py`
  - Added coverage for segment planning, sequential extraction, and AAC segment
    stitching with preserved first priming.
- `tests/test_config_defaults.py`
  - Added defaults for the new SI acceleration switches.

### R1 Decision

- Default final behavior is continuous `aac_mf` where available:
  - no segmented AAC seam;
  - no sidecar `edts` priming (`media_time=0`);
  - real output on `SI_TEST_8K.mp4`: `140958` audio samples,
    `sum_dur=144341016`, `edts=0`, `priming=0`.
