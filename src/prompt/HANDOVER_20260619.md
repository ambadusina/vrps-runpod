## 2DVR Stabilizer Emergency Default-Off

- User real-device retest reported the V1-lite stabilizer was destructive:
  - foreground/object translation was almost locked;
  - soft_shift output showed transparent/blocky garbage inside foreground bodies.
- Diagnosis:
  - The static deadband/max-step limiter can misclassify low-texture foreground
    interiors as static.
  - That locks parts of the near map while the RGB object continues moving.
  - soft_shift then receives inconsistent depth boundaries and its hole/rim
    cleanup can create body-internal artifacts.
- Immediate fix:
  - `PT_TWO_DVR_DEPTH_STABILIZER` / `PT_TWO_DVR_TEMPORAL_DEPTH` now default to
    hard off.
  - `TWO_DVR_TEMPORAL_DEPTH_MODE` now defaults to `off` in config.
  - New disparity-pixel limiter thresholds now default to `0.0`:
    - `PT_TWO_DVR_TEMPORAL_STATIC_DEADBAND_PX=0.0`
    - `PT_TWO_DVR_TEMPORAL_STATIC_MAX_STEP_PX=0.0`
    - `PT_TWO_DVR_TEMPORAL_MOTION_MAX_STEP_PX=0.0`
  - The code remains available as explicit opt-in only.
- CLI usability fix:
  - Offline CLI still lets `--temporal-depth` enable the experimental path;
    when defaults are hard off, the CLI depth mode default is `ema` so
    `--temporal-depth` alone works.
- Verification:
  - Runtime GPU smoke confirmed default renderer state:
    `depth_enabled=False`, `mode=off`, `deadband=0.0`.
  - `.\.venv\Scripts\python.exe -m pytest -p no:cacheprovider tests\test_two_dvr_temporal_stability.py tests\test_config_defaults.py tests\test_two_dvr_ui_hole_fill.py tests\test_two_dvr_hybrid_hole_fill.py -q`

## Offline 2DVR P4 Playback Stutter Investigation

- User reported `videos/test_4k2d.mp4` -> generated 2DVR output plays with
  visible stutter even though bitrate looks normal.
- Actual generated file found:
  - `videos/test_4k2d_S000030_E000330_3D_LR_Screen.mp4`
  - User-mentioned `videos/4k2d_S000030_E000330_3D_LR_Screen.mp4` was not
    present in the workspace.
- Source/output facts from `ffprobe`:
  - Source: `1920x1080`, HEVC Main, ~`59.94fps`, ~`6.25Mbps` video.
  - Output: `3840x1080`, HEVC Main, ~`59.94fps`, ~`17.67Mbps` video,
    `10789` video frames over ~`179.959s`.
  - Flat3D SBS doubles width, so playback workload is roughly `3840x1080@60`,
    regardless of bitrate.
- Strong stutter cause found:
  - `ffmpeg -i videos\test_4k2d_S000030_E000330_3D_LR_Screen.mp4 -map 0:v:0 -f null -`
    reports repeated `Application provided invalid, non monotonically increasing dts`
    warnings.
  - Decoded frame timestamps duplicate or go backward around GOP boundaries
    (about every 60 frames).
  - The file contains B frames. This is risky for the offline 2DVR PyNv path
    because it writes raw HEVC bitstream into an ffmpeg muxer; if reorder timing
    is not represented correctly, MP4 playback can stutter even when bitrate is
    reasonable.
- Fix applied:
  - `offline/two_dvr_pynv.py` now passes `bf=config.PASSTHROUGH_HEVC_BF` to
    `nvc.CreateEncoder`; default is already `0`, matching the realtime PyNv path.
  - This should make newly generated 2DVR PyNv outputs use I/P frames only and
    avoid the observed DTS instability.
- Additional issue found while validating:
  - PyNvVideoCodec expects uppercase P presets (`P4`), but ffmpeg `hevc_nvenc`
    expects lowercase (`p4`).
  - `ui/pages/two_dvr_page.py` now passes lowercase `--preset p4` to the CLI for
    ffmpeg fallback while still setting env `PT_PASSTHROUGH_PYNV_PRESET=P4` for
    PyNv.
- Validation:
  - A 5-second repair probe from the problematic output re-encoded with ffmpeg
    `hevc_nvenc -preset p4 -bf 0` produced no DTS warnings when decoded to null.
  - `ffprobe` on that probe showed I/P frames only in the sampled interval.
  - `.\.venv\Scripts\python.exe -m pytest -p no:cacheprovider tests\test_two_dvr_progress.py tests\test_two_dvr_ui_hole_fill.py tests\test_ui_smoke.py -q`
    -> `9 passed`.
- Existing problematic file is not automatically repaired. It should be
  regenerated after this fix, or re-encoded/remuxed through a timestamp-safe
  path for comparison.
