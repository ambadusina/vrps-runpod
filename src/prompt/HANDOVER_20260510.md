## 2026-05-10 AAC cache miss green-screen slate

- 目标：
  - 解决 live MPEG-TS 首次播放时 AAC cache 构建慢，导致客户端在首包前超时/退出的问题。
  - 只影响 AAC cache miss；cache hit 继续走原 `pipe_ts` 稳定路径。
- 新配置：
  - `PT_PASSTHROUGH_AUDIO_MPEGTS_SLATE=1` 默认开启；
  - 设为 `0` 可回退旧行为。
- 实现：
  - cache miss 时不再同步等待 `_cached_aac_input()`；
  - 启动本地 TCP 音频输入 server；
  - 启动后台 ffmpeg 线程构建 ADTS AAC cache；
  - FFmpeg final mux 从一开始读取两个稳定输入：
    - 视频：同一个 PyNv HEVC encoder 输出；
    - 音频：`tcp://127.0.0.1:<port>` ADTS AAC。
  - 音频 server 起初用 FFmpeg `anullsrc=r=48000:cl=stereo` 生成标准 AAC 静音；
  - AAC cache ready 后，停止静音生成器，将真实 cache 文件继续写入同一个 TCP 连接；
  - 视频侧在 cache ready 前编码同分辨率纯绿 NV12 帧，cache ready 后切换到真实 decode/matting/composite 帧。
- 保留：
  - 同一个 PyNv encoder 实例，避免 HEVC 参数/分辨率变化；
  - MPEG-TS PMT 从一开始就包含视频和音频；
  - cache hit 仍使用 `_open_pipe_ts_muxer(audio_input=cache_path)`。
- 风险：
  - 磁盘 cache 仍是 `-c:a copy`，如果源音频 ADTS copy 本身失败，会回到 cache build failure；
  - 输出侧已统一转码为 48k stereo AAC LC，因此静音垫片和正片进入 mux 的音频规格一致；
  - 如果仍有播放器在静音切正片时异常，下一步应检查 AAC PTS 连续性，而不是先改回磁盘转码。
- 验证：
  - `python -m compileall config.py pipeline\pynv_stream.py` 通过；
  - `git diff --check` 只有 CRLF warning。
- 观察日志：
  - cache miss 应出现 `slate audio server listening`、`slate pipe_ts ...`、`slate video begin`；
  - cache 完成应出现 `slate audio cache ready`、`slate audio switching to cached AAC`、`slate video end`;
  - cache hit 应只出现原 `audio cache hit` 和 `pipe_ts final mux cmd`。

### AAC cache command optimization notes

## 2026-05-10 reference: garrrrido/vr-masking-tools MatAnyone2 usage

Sources checked:
- https://github.com/garrrrido/vr-masking-tools/tree/main
- `pipeline.py` from raw GitHub content

Key findings:
- The reference project does not run MatAnyone2 on the full stereo frame directly.
- Pipeline structure:
  1. Detect intro end and fade transitions.
  2. Split the VR video into left/right eye segments.
  3. Extract each segment's first frame.
  4. Generate first-frame masks with finetuned SAM3.
  5. Run MatAnyone2 separately for left-eye and right-eye segment videos.
  6. Stitch left/right alpha videos back into stereo mask video.
  7. Concatenate segment masks and normalize CFR.
- Important defaults/options from README/pipeline:
  - `--mask-height` default `1280`, allowed `640..1600`, divisible by 16.
  - `--center-crop` crops 10% from each side of each eye, keeping 80% (`CENTER_CROP_RATIO = 0.8`). README claims about 70% faster masking and about 50-55% lower VRAM.
  - `--segment-length` default `10s` in argparse, while helper split default is 5s if called without override.
  - `--warmup` default `15` frames.
  - `--erode` default `3`; `--dilate` default `6` for masks.
  - prompt defaults to `woman` because their SAM3 is finetuned for that target.
- Consequence for our current implementation:
  - We currently feed the full frame through a 512/1024 fixed MatAnyone2 input and bootstrap first mask from RVM.
  - The reference uses much stronger first-frame masks (SAM3) and runs per-eye cropped segments, so MatAnyone2 receives a more focused target at higher effective subject resolution.
  - This likely explains why our full-frame MatAnyone2 output can leave more uncut edge/background than RVM despite using MatAnyone2.
- Potential adaptations for our project:
  - For SBS VR, add an optional offline mode to process each eye independently, then stitch alpha/composite back.
  - Add center-crop per-eye processing before MatAnyone2, with padding back to full eye size.
  - Add user-provided first-frame mask support per eye, or later ONNX detector/SAM-like first-frame mask generation.
  - Add mask morphology controls similar to `erode/dilate`, but apply carefully: MatAnyone2 wants a good first-frame prior, not necessarily final hard matte.

### SAM3 VRAM clarification

- User pointed out correctly:
  - `sam3_decoder.onnx` + `.data` is only a few hundred MB;
  - `sam3_image_encoder.onnx` + `.data` is under about 2GB;
  - therefore the observed 6GB-12GB VRAM is not simply model weight size.
- Clarified in `tools/offline_passthrough.py`:
  - `--sam3-cuda-memory-limit-mb` and `--sam3-decoder-cuda-memory-limit-mb` are ONNX Runtime CUDA arena/workspace cache caps, not model size declarations.
  - SAM3 prepass log now prints `low_memory=... encoder=[CUDAExecutionProvider(arena_cap=...MiB), ...] decoder=[...]`, making it clear whether sessions are being kept resident or loaded/unloaded.
- Current direction:
  - keep SAM3 encoder and decoder resident on GPU by default (`--no-sam3-low-memory`, currently default);
  - process one sampled eye/frame at a time and drop NumPy/ORT output references after decode;
  - reserve `--sam3-low-memory` only as a slow fallback when arena/workspace peak cannot fit.
- User test showed resident GPU mode with encoder arena cap `6144MiB` and decoder arena cap `3072MiB` failed in image encoder `Transpose node_permute_32`:
  - ORT reported only about `16MB` available inside the encoder arena while needing about `21MB`;
  - this means the per-session arena cap was too tight, not that the full GPU was out of memory.
- Added a narrow retry:
  - on ORT CUDA/BFCArena allocation failure, recreate only the exhausted SAM3 session and retry once;
  - normal resident mode still keeps encoder/decoder loaded and does not unload every sample.
- User decided SAM3 should be efficiency-first and restricted to 16GB VRAM users instead of trying to fit 8GB GPUs.
- Updated defaults after testing/feedback:
  - `--sam3-cuda-memory-limit-mb 8192`;
  - `--sam3-decoder-cuda-memory-limit-mb 4096`;
  - fully uncapped arenas can consume over 15GB on a 16GB GPU and become slower due low free-memory headroom;
  - the defaults are intended as a 16GB efficiency-first cap, not an 8GB compatibility mode.
- Added `--sam3-min-vram-gb`, default `15.5`:
  - SAM3 CUDA prepass checks total GPU VRAM before starting;
  - use `15.5` because 16GB cards often report about `15.9GB` through `nvidia-smi`;
