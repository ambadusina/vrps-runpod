## 2026-05-18 MatAnyone2 Medium Low-VRAM Plan/Changes
- Investigated reducing MatAnyone2 medium peak VRAM for 8GB and ideally sub-6GB GPUs.
- ONNX stages in medium mode:
  - prepass: `yolov8s-worldv2.onnx` + `efficientsam_s.onnx` + `models/person_txt_feats.npy`
  - main propagation: MatAnyone2 `matanyone2_image_key.onnx`, `matanyone2_mask_memory.onnx`, `matanyone2_first_frame_refine.onnx`, `matanyone2_step_update.onnx` from the selected `matanyone2_onnx_*` folder
  - RVM ONNX is not loaded for inference in MatAnyone2 mode; `Matter(load_model=False)` is used only for NV12 upload/preprocess/composite helpers.
- Previous medium mode kept YOLO/EfficientSAM CUDA sessions resident in the same process before loading MatAnyone2; this could stack ORT CUDA arenas and push peak VRAM over 8GB.
- Added YOLO/EfficientSAM prepass subprocess support:
  - new hidden `--ywes-prepass-out`
  - default `--ywes-subprocess` enabled
  - prepass writes masks/segment starts to NPZ, exits, and releases its CUDA context before MatAnyone2 loads
- Changed `offline/convert.py` so `matanyone2_medium` now passes `--matanyone2-batch 1` by default. This forces MatAnyone2 `512_bs1` and processes left/right eyes sequentially, lowering peak memory versus batch2.
- Verified command mapping now emits `--matanyone2-prepass yoloworld_efficientsam --matanyone2-batch 1`.
- Ran UI-environment-simulated 1s low-memory smoke:
  - output `debug_output\test_4k_matanyone2_medium_1s_lowmem.mp4`
  - prepass ran in subprocess with CUDA, one sample about `0.9s`
  - main process loaded `models\matanyone2_onnx_512_bs1`, log showed `batch2=False`
  - conversion succeeded; throughput about `13.5 fps`
- Remaining low-VRAM levers:
  - keep `PT_PASSTHROUGH_PYNV_DECODER=simple`, threaded decoder buffer/batch already defaults low (`batch=1`, `buffer=2`)
  - use `--alpha-stride 2` or `3` for fewer MatAnyone2 updates, reducing transient work and improving speed, but it does not remove the main session weights
  - reduce active segment frequency only affects prepass frequency, not main MatAnyone2 session peak
  - if sub-6GB is still not met, next likely step is exporting/using FP16 MatAnyone2 ONNX; current manifests show `fp16=false`
- Mis-detection/left-right mismatch analysis:
  - YOLO-World with only `person` txt_feats can still pick the highest-scoring non-person-like region because scores are weak/open-vocabulary similarities, not COCO-class calibrated confidences
  - left/right eyes are processed independently; different crops/parallax/occlusion can select different boxes even deterministically
  - there is no intended randomness in inference; differences come from independent detection/NMS and low score margins
  - mitigation candidates: use left-eye detection projected to right eye when scores are close, add temporal/segment box smoothing, require area/aspect-ratio constraints, or use EfficientSAM overlap/area sanity checks before accepting a detection
