## MatAnyone2 SBS Left-Eye Drag Asymmetry Fix

- User reported SBS output where the left eye still had severe drag while the right eye was much cleaner.
- Root cause accepted from review and verified in code:
  - `Matter._ensure_dev_chw()` / `_ensure_pinned_chw()` reuse shared batch buffers.
  - The MatAnyone2 single-eye path preprocessed left and right first, then ran left and right. With `batch=1,batch_idx=0`, the right-eye preprocess overwrote the left-eye image buffer before the left-eye model run.
  - This affected both IOBinding device views and non-IOBinding pinned host views.
- Implemented fix in `offline/matanyone2_engine.py`:
  - `_preprocess_eye()` now uses `batch=2,batch_idx=eye_idx` when an eye index is known.
  - `_preprocess_eye_roi()` applies the same per-eye batch slot rule.
  - IOBinding output OrtValue slots are now separated per eye via `_step_io_outputs[eye_idx]` and `_step_io_slots[eye_idx]`, preventing same-eye input/output aliasing across frames.
- Added unit coverage in `tests/test_matanyone2_engine.py`:
  - output OrtValue buckets are independent per eye;
  - normal and ROI preprocess calls pass `batch=2,batch_idx=eye_idx`.
- Initial verification:

## YOLO26m fp16+CUDA Silent-Collapse Root Cause and Fix

- User reported that processing 30s of `videos/72456_3840p.mp4` with the new
  `yolo26m_efficientsam` prepass detected no person at the head and tail of the
  clip while middle samples worked. Symptom looked like edge-of-clip recall
  loss; root cause was actually broken model outputs on most timestamps.
- Wrote `tools/probe_yolo26m_real.py` to ingest one frame at a chosen timestamp
  and dump top sigmoid scores per preprocessing assumption (`div255`,
  ImageNet-normalized, BGR). For `videos/72456_3840p.mp4` at t=0.0s the
  per-eye person sigmoid maxed at ~0.88 with the plain `/255` preprocessing,
  confirming the masker code's preprocess pipeline is correct.
- Wrote `tools/probe_y26es_prepass.py` to drive the production
  `Yolo26mEfficientSamMasker` end-to-end on real SBS frames at 10 evenly
  spaced timestamps. With the original default
  `--y26es-yolo-model yolo26m_model_fp16.onnx` on CUDA:
  - t=0.00s and t=30.00s returned `raw detect L=0 R=0`;
  - t=3.33s..t=26.67s all detected with scores 0.62-0.93.
- Cross-matrix of provider x weights at t=0.0s:
  - CPU + fp32: 0.88
  - CPU + fp16: 0.86
  - CUDA + fp32: 0.88
  - CUDA + fp16: **0.011** (broken; all 300 queries collapse)
- Root cause: `models/yolo26m/yolo26m_model_fp16.onnx` outputs collapse to
  near-zero person sigmoid when loaded with ORT CUDAExecutionProvider. ORT
  also emits `4 Memcpy nodes added to ... CUDAExecutionProvider` and
  `ScatterND with reduction='none'` warnings during init.
- Fix landed:
  - Default `--y26es-yolo-model` changed from `yolo26m_model_fp16.onnx` to
    `yolo26m_model.onnx` in `offline/yolo26m_efficientsam.py:92`,
    `offline/yolo26m_efficientsam.py:488`,
    `tools/offline_alpha_passthrough.py:962`, and
    `tools/offline_passthrough.py:998`.
  - `Yolo26mEfficientSamMasker.__init__` now runs a synthetic mid-gray
    sanity probe whenever the loaded weights look fp16 and CUDA is active;
    a max sigmoid below 0.05 triggers a loud `WARNING` line telling the user
    to switch back to the fp32 file. Verified: the fp16+CUDA path produces
    `max sigmoid=0.0236` on the probe and the warning fires.
- Verification after fix:
  - `probe_y26es_prepass.py` on the default model now detects at every
    sampled timestamp 0..30s, scores 0.83-0.93, `stereo: mode=paired` at
    every frame.
  - `pytest tests/test_yolo26m_efficientsam.py tests/test_matanyone2_engine.py tests/test_offline_convert.py tests/test_matanyone2_trt_runtime_paths.py tests/test_i18n.py tests/test_ui_smoke.py`
    -> 41 passed, 4 subtests passed.

## YOLO26m L/R Mask Asymmetry Fix (test_8k_3.mp4)

- User reported large L/R mask differences on `videos/test_8k_3.mp4` first 30s.
- `tools/probe_y26es_prepass.py` surfaced two distinct failure modes:
  1. **Cross-eye box-size mismatch (t=2.00s)**: L returned a 0.37-area box
     covering person + sofa while R returned a 0.12-area box on the person
     alone (3.0x area ratio). Stereo cost still tagged them `paired` because
     box centers matched, so the masks ended up 2x apart in area
     (L=0.1475 vs R=0.0726).
  2. **Fill-the-frame fallback (t=20-24s)**: both eyes returned 0.58-0.62
     area boxes (failed plausibility 0.55 upper bound) and the
     `fallback_score_gate` path accepted them anyway, yielding mask area
     ~0.38 on both eyes (whole frame painted green).
- Implemented in `offline/yolo26m_efficientsam.py`:
  - Added `max_box_area=0.50` and `cross_eye_area_ratio=1.5` params on the
    masker (and matching CLI args `--y26es-max-box-area` and
    `--y26es-cross-eye-area-ratio` on both tools).
  - Plausibility area upper bound switched from hard-coded 0.55 to
    `self.max_box_area` (default 0.50).
  - After a successful `paired` selection, if L/R area ratio exceeds
    `cross_eye_area_ratio`, project the higher-score side to the other eye
    via the existing `_project_detection`; reports `mode=paired_project_l2r`
    or `paired_project_r2l`.
  - `fallback_score_gate` path now also enforces `_is_within_max_box_area`
    so "fill the frame" candidates can no longer slip through after
    plausibility rejects them.
- Tests added in `tests/test_yolo26m_efficientsam.py`:
  - `test_cross_eye_area_mismatch_projects_higher_score_side`
  - `test_max_box_area_rejects_fill_the_frame_in_plausibility_and_fallback`
  - Updated existing plausibility boundary test to the new 0.50 cap.
- Verification after fix on `videos/test_8k_3.mp4` (probe with 16 samples
  across 30s):
  - t=2.00s: `paired_project_r2l`, L area_ratio=0.0732 vs R=0.0726
    (delta < 1%, mask is symmetric and tight on person only).
  - t=20.00s/22.00s/24.00s: `no_detection` (whole-frame boxes correctly
    rejected; gap_fill will reuse t=18s or t=26s mask).
  - All other timestamps unchanged.
- Regression:
  - `pytest tests/test_yolo26m_efficientsam.py tests/test_matanyone2_engine.py tests/test_offline_convert.py tests/test_matanyone2_trt_runtime_paths.py tests/test_i18n.py tests/test_ui_smoke.py`
    -> 43 passed, 4 subtests passed.
  - `py_compile` clean on the modified files.
- Follow-up suggestions (not done):
  - Watch for false-negatives at t=20-24s style frames: now they return no
    detection. If real extreme-close-up content with arms spread legitimately
    needs box area > 0.50, the user can pass
    `--y26es-max-box-area 0.60`. The cross-eye projection guard is unchanged.
  - On 8K SBS, YOLO26m@640 input shows its limits; if asymmetry comes back
    on other clips, consider lowering `--y26es-cross-eye-area-ratio` to 1.3.

## YOLO26m Multi-Person Stereo Pairing (sub_35s.mp4)

- User-reported symptom: `videos/sub_35s.mp4` has two people in the scene
  but YOLO26m+EfficientSAM only ever produced one mask per eye (output
  `sub_35s_matanyone2m_S000000_15S_LR_180_SBS_passthrough.mp4`). The
  prepass log showed every frame as `objects=1 cand=L2-4/R2-4 plaus=L2/R2`
  — model and plausibility filter both saw 2 persons per eye, but the
  pipeline only kept 1.
- Two-layer limitation in the original code:
  1. `--y26es-top-k` defaulted to `1`, so the post-NMS slice already
     truncated detections to 1 per eye.
  2. **Even with `--y26es-top-k=2`**, `select_stereo_detections()`
     hard-coded "find single best (l, r) pair" via a flat
     argmin loop and wrote `left_sel = [best_pair[0]]` /
     `right_sel = [best_pair[1]]`. So raising the CLI flag had no
     downstream effect — still 1 person.
- Refactored `offline/yolo26m_efficientsam.py::select_stereo_detections`
  to greedy multi-pair selection:
  - Extracted pair-cost into `_pair_cost()` (same formula as before:
    `4|Δcy| + 2|Δh| + |Δaspect| + 2|Δarea| - 4(score_l + score_r)`).
  - Outer loop iterates up to `self.top_k` times; each iteration picks
    the lowest-cost (l, r) pair among unused candidates, with one-use
    constraint via `used_l` / `used_r` index sets. Loop stops if no
    candidate remains on either side before `top_k` reached.
  - Per-pair cross-eye-area-ratio projection (the L/R asymmetry fix)
    is applied to each pair independently, so a healthy first pair and
    a mismatched second pair coexist in the same scan point.
  - Added `_stereo_paired_mode()` helper that maps per-pair projection
    directions (`l2r` / `r2l` / `none`) to a single mode label. Single-
    pair behavior is unchanged (`paired`, `paired_project_l2r`,
    `paired_project_r2l`); when multiple pairs have heterogeneous
    projections, the label becomes `paired_project_mixed`.
  - Single-eye projection branches (`project_left_to_right` /
    `project_right_to_left`) and the score-gate fallback now also
    iterate up to `top_k`.
  - Info dict gained `pairs` (count of returned pairs) and
    `projection_dirs` (per-pair direction list) so the prepass log line
    remains useful when multiple persons are tracked.
- Bumped default `--y26es-top-k` from `1` to `2` in
  `tools/offline_alpha_passthrough.py`, `tools/offline_passthrough.py`,
  the `Yolo26mEfficientSamMasker.__init__` class default, and the
  `getattr` fallback in `precompute_segment_masks`. Cost on a true
  single-person clip is negligible because plausibility filters down
  to 1 candidate before EfficientSAM is even called for the second.
