## 2026-05-07 output FPS cap after decode measurements

- User measured pure decode throughput:
  - `test_4k.mp4 --max-side 4096`: about `55.85 fps`;
  - `test_8k.mp4 --max-side 4096`: about `54.14 fps`;
  - 8K at original raw output resolution: only about `7 fps`.
- Conclusion: 59.94fps sources cannot be treated as a full-frame realtime target even before matting/encoding.
- `PT_ALPHA_STRIDE` only reduces matting inference count; decode, composite, and encode still happen for every output frame.
- Added a true output FPS cap:
  - `config.PASSTHROUGH_MAX_FPS`;
  - env `PT_PASSTHROUGH_MAX_FPS`;
  - default `0` means keep source FPS.
- `DecoderProcess` appends an `fps=fps=<cap>:round=near` filter in the output chain.
- `out_info.fps` now reflects capped output FPS, and `EncoderProcess` uses `dec.out_info.fps`, so muxed output rate drops consistently to 30fps when capped.
- `tools/bench.py pipeline/transcode/decode/decode-matrix` added or wired `--max-fps`.
- Target frame counts, summaries, and realtime verdicts now use decoder output FPS rather than source FPS.
- Note: FFmpeg FPS filtering reduces rawvideo pipe, matting, composite, and encode frame count, but may not fully halve hardware decode cost for long-GOP/reference-frame video.
- If 30fps remains unstable, use `decode-matrix --max-fps 30` to split whether the bottleneck is NVDEC, raw pipe, or later stages.
- Verification: `python -m py_compile config.py main.py pipeline\ffmpeg_io.py tools\bench.py pipeline\stream.py` passed.
- Later retest: `tools/bench.py transcode ... --max-fps 30` was only about `21.9 fps`, while `--max-fps 0` was about `23.8 fps`.
- This showed the bottleneck was still mostly matting+encode, not decoder output FPS.
- `run_server.bat` default cap was rolled back to `0` at that time, keeping the cap as an explicit experiment switch.
- User quality constraint:
  - 2:1 VR videos must keep `PT_MATTING_SPLIT_SBS=1` to reduce multi-person/left-right-eye cross-talk issues;
  - matting quality should keep at least `MATTING_INPUT_SIZE=512`;
  - do not recommend `320/384` or `split_sbs=0` as defaults.
- Historical performance references are only for cost understanding:
  - `split_sbs=0` with aspect-preserving whole-SBS input is faster, for example `(1,3,384,768)` or `(1,3,320,640)`, around `29.85/30.22 fps`;
  - this is not the target quality tier.

## 2026-05-07 optimization plan: HQ baseline without alpha skipping

- New baseline constraints:
  - no TensorRT;
  - `PT_MATTING_SPLIT_SBS=1`;
  - `PT_MATTING_INPUT_SIZE=512`;
  - `PT_ALPHA_STRIDE=1`;
  - `PT_PASSTHROUGH_MAX_FPS=0`;
  - 4K SBS NV12 input/output.
- Baseline results in `baseline/baseline_20260507.txt`:
  - `pipeline`: about `21.40 fps`, `avg_preprocess=2.12ms`, `avg_ort_run=32.14ms`, `avg_composite=9.77ms`;
  - `transcode`: about `16.97 fps`, `avg_matting=58.08ms`, `avg_enc_write=13.64ms`.
- Assessment:
  - decode is no longer the main bottleneck;
  - matting hot path without encoding is about `44ms`;
  - with NVENC enabled, transcode matting rises to about `58ms`;
  - likely causes are ORT/CuPy/NVENC GPU contention and synchronization cost.
- Plan:
  1. Add diagnostics and low-risk switches first: make `tools/bench.py transcode` split `pre/ort/comp`, and expose `PASSTHROUGH_PRESET/BITRATE/GOP` plus NVENC tune/rc as env vars for A/B tests (`p1/p2/p4`, `-tune ull`, `cbr_ld_hq`).
  2. Align bench and real stream path: make transcode bench use the same pinned NV12 output pool as `PassthroughStream`.
  3. Core optimization: RVM CUDA IOBinding. Bind CuPy CHW GPU preprocess output directly to ORT and keep `src/r1i-r4i/downsample_ratio` and `fgr/pha/r1o-r4o` on GPU as much as possible, avoiding `_g_chw -> pinned host -> ORT H2D -> alpha D2H -> CuPy H2D` round trips.
  4. Enable CUDA Graph after IOBinding is stable: fixed `shape=(2,3,512,512)`, try CUDA EP `enable_cuda_graph=1` to reduce kernel launch / CPU scheduling cost.
  5. Share the CUDA stream between CuPy kernels and ORT via CUDA EP `user_compute_stream` to reduce cross-stream synchronization.
  6. Add GPU alpha cache after IOBinding so stride/reuse does not download alpha to CPU and upload it again.
- Expected benefit order:
  - IOBinding largest;
  - CUDA Graph next;
  - NVENC settings low-risk incremental gain;
  - shared stream and GPU alpha cache are cleanup after IOBinding.

## 2026-05-07 code review and matting model experiments

- This handoff focused on reviewing `pipeline/stream.py` and `pipeline/matting.py` after GPU optimization work.
- Recent profiling showed decode wait was absorbed by the prefetch thread; the main bottleneck was MODNet ORT inference.
- In SBS mode, left and right halves had previously run separate ORT calls.
- Reviewed the `PassthroughStream` thread chain:
  - decoder thread prefetches raw frames;
  - producer thread runs matting/composite and writes encoder input;
  - reader thread delivers encoder stdout into an asyncio queue.
- Reviewed the `Matter` hot path:
  - upload full frame to CuPy device;
  - GPU preprocess to CHW pinned host input;
  - ORT inference returns low-res alpha;
  - fused CUDA kernel performs upsample + green composite;
  - output returns through pinned host memory.
- Added SBS batch=2 ORT support in `pipeline/matting.py`:
  - reads `input_meta.shape` at init to determine whether batch dimension is dynamic or at least 2;
  - if the model supports batch=2, CPU SBS fallback combines left/right halves into `(2,3,H,W)` for one ORT call;
  - GPU hot path allocates batch-aware device/pinned CHW buffers and writes each half into one batch slot before one ORT call;
  - if model metadata is fixed batch=1, keeps old logic and runs one ORT call per half;
  - logs `shape=... batch2=...` so target machine startup confirms the real path.
- Added visible diagnostics:
  - `Matter` init prints `model input=... shape=... type=... batch2=... providers=...` directly to stderr;

Conclusion:

- TensorRT is not the cause; active providers are already CUDA/CPU.
- PyNvVideoCodec is not the cause; pure ORT reproduces the cold-start delay.
- IOBinding is not the cause; disabling it still has ~175s first ORT.
- Shared CUDA stream is not the cause.
- `cudnn_conv_algo_search=HEURISTIC` did not improve cold start in this environment.
- Cold-start cost is isolated to the first CUDA EP execution of the RVM graph on this machine/runtime.

Working interpretation:

- This is likely ONNX Runtime CUDA EP / CUDA 12.9 / RTX 5060 Ti first-execution initialization or kernel/cudnn path cost.
- The correct product mitigation is startup warmup plus possibly a persistent Matter singleton, not request-time first inference.
- Because steady-state is excellent, this does not invalidate the PyNv backend direction.
