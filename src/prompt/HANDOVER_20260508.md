## Phase 1.1 PyNv encoder probe

Goal:

- Validate PyNvVideoCodec NVENC API independently from the server.
- Confirm CUDA NV12 input can be encoded to raw H.264 Annex-B.

Added:

- `tools/pynv_encode_probe.py`
  - Generates synthetic NV12 frames.
  - Supports CPU input (`--cpu-input`) and GPU input.
  - GPU input uses an AppFrame-style object with `cuda()` returning Y and UV CUDA Array Interface planes.
  - Writes raw `.h264`.
  - Runs `ffprobe` on the H.264 output.

Important API finding:

- `CreateEncoder(width, height, "NV12", usecpuinputbuffer, ...)` requires the pixel format as a string, not `Pixel_Format.NV12`.
- Direct CuPy ndarray input failed.
- GPU input works when wrapped as an object with:

```python
def cuda(self):
    return [y_plane_cai, uv_plane_cai]
```

PTS / VFR finding:

- PyNvVideoCodec 2.1.0 `Encode()` exposed signatures are:

```text
Encode(frame) -> bytes
Encode(frame, picFlags) -> bytes
Encode(frame, picFlags, SEI) -> bytes
EndEncode() -> bytes
```

- No exposed `pts`/`dts` parameter was found.
- The encoder returns raw H.264 bytes, not timestamped packets.
- The current path feeds FFmpeg with:

```text
ffmpeg -f h264 -framerate <fps> -i - -c:v copy ...
```

- This input method has no per-frame PTS side channel. FFmpeg must synthesize CFR timestamps from `-framerate`.

Product implication:

- Strategy C (`PTS passthrough + frame dropping by source PTS`) cannot be fully implemented on the current "raw H.264 bytes -> FFmpeg stdin" mux path.
- For this path, the realistic pre-Phase-2 strategy is CFR output at the configured passthrough fps.
- If exact VFR/source PTS preservation is required, the mux layer must change to an API that accepts timestamped packets (for example libav/PyAV or PyNv mux/transcoder APIs if they expose packet timestamps).

Current recommendation:

- Continue Phase 2 as an explicit CFR passthrough backend first, matching current DLNA 30fps product target.
- Track exact PTS/VFR preservation as a separate post-Phase-2 requirement.
- Before enabling PyNv by default, test real-device behavior with CFR 30 output and explicit color metadata.

## 2026-05-08 CFR effective FPS fix for PyNv probes

Problem:

- PyNvVideoCodec 2.1.0 raw H.264 encode path has no exposed per-frame PTS/DTS.
- FFmpeg mux therefore synthesizes timestamps from `-framerate`.
- A hardcoded `-framerate 30` is only correct for 30fps output.
- For 24fps CFR sources it compresses duration; for 60fps uncapped sources it doubles duration.

Added:

- `utils/video_metadata.py`
  - `VideoTimingMetadata`
  - `parse_rate()`
  - `probe_timing_metadata(path)`
  - `cfr_source_index(out_index, source_fps, output_fps)`
  - CFR detection from `r_frame_rate` vs `avg_frame_rate` with `<1%` relative difference.
- `tools/pynv_transcode_probe.py`
  - `--fps` now means max/cap fps.
  - Computes `effective_fps = min(source_fps, cap_fps)` when cap is set.
  - Uses `effective_fps` for PyNv encoder fps and FFmpeg mux `-framerate`.
  - Selects decoded source indices by CFR ratio instead of reading `0..target-1`.
  - Prints source/effective fps, CFR status, and source index span.
- `tools/pynv_fullchain_probe.py`
  - Same effective-fps and source-index selection logic.

## 2026-05-08 Passthrough pseudo-VOD byte seek integration

Goal:

- Make DLNA/progressive clients classify passthrough as seekable by avoiding chunked-only semantics.
- Preserve the project constraint that realtime matting output is not byte-deterministic; byte Range is mapped to estimated time, not exact file offsets.

Implemented:

- `utils/bitrate_estimator.py`
  - Stores estimates in `runtime_cache/bitrate_estimates.json`.
  - Uses `PT_PASSTHROUGH_BITRATE` on first play.
  - Adds a safety multiplier and container overhead to deliberately overestimate pseudo `Content-Length`.
  - Records measured session bitrate from emitted stream bytes.
- `config.py`
  - `PT_SEEK_MODE` now defaults to `bytes`.
  - Added `PT_PASSTHROUGH_MAX_CONCURRENT`, default `1`.
