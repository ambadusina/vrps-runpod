# HANDOVER 2026-05-06

## Background

- The project is a DLNA realtime passthrough green-screen transcoding server.
- The original hot path was:
  - FFmpeg decode -> MODNet matting -> FFmpeg encode -> HTTP stream.
- The initial sample `videos/test.mp4` was HEVC `yuv420p10le`, `2048x2048`, about `59.94fps`.
- Software decode throughput was clearly below realtime.

## Initial decoder diagnostics

- Enhanced `tools/bench.py decode/pipeline` diagnostics:
  - prints source `codec_name` and `pix_fmt`;
  - prints the final selected decoder strategy;
  - prints output size and full FFmpeg command;
  - prints failed strategies and stderr summaries.
- `pipeline/ffmpeg_io.py` `probe()` now returns `codec_name` and `pix_fmt`.
- Added CUDA target-machine decoder candidates:
  - `*_cuvid + -resize`;
  - HEVC tries `hevc_cuvid+resize` first when available.
- Fixed the 10-bit CUDA hwdownload filter chain:
  - chooses `p010le` or `nv12` based on source `pix_fmt`;
  - then scales/formats to `bgr24`.
- Fixed `pipeline/stream.py` so the encoder uses `dec.out_info` dimensions/fps after downscale, instead of source video dimensions.

## Follow-up records

- Target-machine logs showed `hevc_cuvid+resize` selected, outputting `1024x1024 bgr24`, but still only about `14fps`.
- This suggested the bottleneck was no longer HEVC bitstream decode, but likely one of:
  - NVDEC resize download;
  - pixel format conversion to `bgr24`;
  - Python rawvideo pipe reading.
- Added `tools/bench.py decode-matrix` to split decode costs:
  - `cuvid-null`;
  - `cuvid-yuv420p-devnull`;
  - `cuvid-bgr24-devnull`;
  - `python-pipe-bgr24`.
- Changed `DecoderProcess.read_frame()` to `readinto(bytearray)` to reduce large Python bytes concatenation/copy overhead.
- Fixed `tools/bench.py pipeline` reshaping frames with source dimensions after decoder downscale.
- New target-machine `decode-matrix` logs showed:
  - `cuvid-bgr24-devnull` about `395fps`;
  - `python-pipe-bgr24` about `215fps`.
- Pure decode was no longer the bottleneck.
- FFmpeg rawvideo logs showed `dup=59`; added `-fps_mode passthrough` in decoder output and `decode-matrix` to avoid CFR duplicate-frame work.
- Target retest reached about `215fps` for `decode`; `pipeline` still about `11.94fps`, moving the bottleneck to MODNet matting/composite.

## Matting profiling and optimizations

- Reworked `pipeline/matting.py` while preserving `Matter.composite_green()` API.
- Added:
  - `alpha_profile()`;
  - `composite_green_profile()`;
  - timing split for `preprocess / ort / alpha_resize / composite`.
- Optimized green-screen composite:
  - avoided allocating a full green background frame;
  - avoided 3-channel float alpha expansion;
  - reused a single-channel float alpha buffer and composed per channel.
- `tools/bench.py pipeline` now reports average `pre / ort / alpha_up / comp` timings.
- Added `tools/bench.py matting` to repeatedly test the pure MODNet matting path on one frame.
- Target-machine profiling for `1024` output and `512` MODNet input reached about `27fps`, with steady-state around:
  - `pre=4ms`;
  - `ort=15ms`;
  - `alpha_up=1ms`;
  - `comp=10ms`.
- CPU composite for `1024x1024` was already close to the half-frame budget for 60fps.

## DLNA seek / progress follow-up

- Stopped pretending passthrough `Content-Length` equals the original source file size.
- Rationale: after realtime matting, container, bitrate, and frame dimensions can differ; fake byte length misleads player Range, buffering, and EOF decisions.
- `/passthrough/{name}` now parses `TimeSeekRange.dlna.org` request headers:
  - supports `npt=120.5-`;
  - supports `npt=00:02:00-`;
  - parsed time overrides `?t=` and starts `PassthroughStream` from that second.
- `/passthrough/{name}` logs now include:
  - `requested_t`;
  - `TimeSeekRange.dlna.org`;
  - `Range`;
  - `getcontentFeatures.dlna.org`;
  - `transferMode.dlna.org`.
- Passthrough responses return `TimeSeekRange.dlna.org` and `X-AvailableSeekRange.dlna.org` when source duration is known, while keeping `Accept-Ranges: none` because transcoded streams have no reliable byte Range mapping.
- Verified: `python -m py_compile http_app\routes_media.py` passed.
- Phone logs still showed GET `/passthrough` with `time_seek=None range=None`; the client did not treat the resource as seekable.
- Updated ContentDirectory passthrough item from `DLNA.ORG_OP=00` to `DLNA.ORG_OP=10` to advertise DLNA time seek only.
- `DLNA.ORG_CI=1` marks it as transcoded.
- Flags changed to `417000...` for time-based seek advertisement.
- ContentDirectory switches passthrough `mime` / `DLNA.ORG_PN` to `video/MP2T` / `AVC_TS_HD_60_L2_T` when `PT_CONTAINER=mpegts`.
- `/passthrough` GET/HEAD `contentFeatures.dlna.org` matches `OP=10;CI=1;FLAGS=417000...`.
- Added `HEAD /passthrough/{name}` to return only DLNA/seek metadata without starting FFmpeg/MODNet.
- Added `video/MP2T` to `ConnectionManager.GetProtocolInfo`.
- Verified: `python -m py_compile http_app\routes_media.py http_app\routes_dlna.py dlna\content_directory.py` passed.

## Shutdown and byte-seek fixes

- Found Ctrl+C could leave Uvicorn stuck at `Shutting down`.
- Suspected cause: `iter_bytes()` used an executor to block-read encoder stdout, while `EncoderProcess.close()` only closed stdin and did not close stdout or terminate the active FFmpeg process.
- Improved `DecoderProcess.close()` / `EncoderProcess.close()`:
  - close all pipes;
  - terminate and short wait;
  - kill and wait if needed.
- `PassthroughStream.close()` joins producer thread for 1 second and logs if it does not exit.
- Verified: `python -m py_compile pipeline\ffmpeg_io.py pipeline\stream.py` passed.
- Phone still did not show seek; GET logs showed `time_seek=None range=None`.
- Added `PT_SEEK_MODE`:
  - default `time` keeps old behavior;
  - experimental `bytes` advertises `DLNA.ORG_OP=11` / `FLAGS=617000...` in ContentDirectory and GET/HEAD responses.
- Bytes mode estimates Content-Length from `PASSTHROUGH_BITRATE * duration / 8` and returns `Accept-Ranges: bytes`.
- Incoming `Range: bytes=start-` maps approximately to `start / estimated_size * duration` and returns 206.
- Bytes mode is only a compatibility experiment; generated output size is not guaranteed to match the estimate.
- Verified: `python -m py_compile config.py main.py http_app\routes_media.py dlna\content_directory.py` passed.
- First bytes-mode implementation also returned estimated `Content-Length` in streaming GET responses.
- That caused Starlette `RuntimeError: Response content shorter than Content-Length` when clients disconnected or TS output was shorter than estimated.
- Fixed: estimated `Content-Length` is returned only from `HEAD /passthrough`; real `GET` omits `Content-Length`, while retaining `Accept-Ranges: bytes`, `Content-Range`, and approximate range-to-time mapping.
- Verified: `python -m py_compile http_app\routes_media.py` passed.
